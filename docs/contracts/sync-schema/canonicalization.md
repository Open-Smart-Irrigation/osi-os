# Sync Payload Canonicalization

This document pins the canonical form used to compute payload hashes for sync events. All runtimes that compute or verify hashes must produce byte-identical output for the same logical payload.

## Algorithm

1. Project the event payload to its canonical body (transport metadata excluded — no batch wrapper, no auth context, no receipt timestamp).
2. Normalize per the rules below.
3. Serialize as compact JSON (no insignificant whitespace).
4. Encode UTF-8.
5. Hash with SHA-256.
6. Express as lowercase hex (64 chars).

## Rules

### Object keys
Sorted lexicographically by UTF-16 code units, matching Java `String.compareTo`
and ECMAScript's default string sort. Stable across nested objects.

### Whitespace
None. No spaces between tokens. No trailing newlines.

### Strings
UTF-8. Standard JSON escape: `\"`, `\\`, `\b`, `\f`, `\n`, `\r`, `\t`, and `\uXXXX` for code points < 0x20. Do not use `\uXXXX` for printable ASCII or for code points >= 0x20 that JSON permits literally.

Normalization is recursive and content-based. Every string value is normalized
when its entire contents match a UUID, EUI-48, EUI-64, or complete
offset-timestamp candidate, regardless of its property name or nesting depth.
Property names do not control recognition. Substrings and local timestamps
without an offset remain unchanged.

### Timestamps
A complete timestamp candidate has the form
`YYYY-MM-DDTHH:MM:SS[.<digits>](Z|±HH:MM|±HHMM)`, where `<digits>` is a
nonempty decimal sequence. Fractions of 1–9 digits are accepted. Accepted
candidates are converted to UTC and serialized with fixed millisecond
precision: `YYYY-MM-DDTHH:MM:SS.sssZ`. Sub-millisecond precision is truncated,
not rounded. A complete candidate with more than nine fractional digits is
invalid; it is neither truncated nor preserved as text.

### Numbers
Fixed-point decimal string. No exponent form. No negative zero.
- Integers: bare digits, no trailing `.0`. Example: `42`, not `42.0`.
- Floats: preserve the producer's finite decimal value, serialize in fixed-point form, and trim trailing zeros. Do not round or truncate merely to fit a digit limit; values with more than 9 significant digits remain valid if the runtime can represent them exactly enough to round-trip. Example: `3.14`, `0.000001`, `1.5`.
- Never `NaN`, `+Infinity`, `-Infinity`; reject the payload at canonicalization time if encountered.

### Null vs absent
`null` is preserved. Absent fields stay absent — never serialized as `null`.

### UUIDs
Lowercase, hyphenated 8-4-4-4-12. Example: `d4fe4b8f-2c58-4d1c-a8c3-9dce37e6ec90`.

### EUIs
Uppercase, no separators. Example: `0016C001F11715E2`. EUI-48 input is expanded with the standard `FFFE` rule before canonicalization.

### Booleans
`true` and `false` lowercase, as standard JSON.

## Test Vectors

The following pairs are normative. A runtime is conformant if its hash matches the expected value for every input.

### Vector 1 — empty object
- Input: `{}`
- Canonical: `{}`
- SHA-256: `44136fa355b3678a1146ad16f7e8649e94fb4fc21fe77e8310c060f61caaff8a`

### Vector 2 — sorted keys
- Input: `{"b": 1, "a": 2}`
- Canonical: `{"a":2,"b":1}`
- SHA-256: `d3626ac30a87e6f7a6428233b3c68299976865fa5508e4267c5415c76af7a772`

### Vector 3 — timestamp precision
- Input: `{"recorded_at": "2026-05-03T12:00:00.123456789Z"}`
- Canonical: `{"recorded_at":"2026-05-03T12:00:00.123Z"}`
- SHA-256: `b2433460d6039d8b8bb640cb46d854e5fcdc6c511a4152ba0f9add660ceea9cc`

### Vector 4 — float fixed-point
- Input: `{"flow_rate_lpm": 8.5e0}`
- Canonical: `{"flow_rate_lpm":8.5}`
- SHA-256: `890beb02858df19a0d452b7a3bac58118a180b2e61d41e810d10768c18acb727`

### Vector 5 — null preserved, absent dropped
- Input: `{"a": null, "b": null}` with field `c` absent
- Canonical: `{"a":null,"b":null}`
- SHA-256: `052c4bd5e6ded53bd884485af8b1667a7b70ba3a8573b54bd878f6d2c705c2df`

### Vector 6 — EUI uppercase, UUID lowercase
- Input: `{"device_eui": "0016c001f11715e2", "event_uuid": "D4FE4B8F-2C58-4D1C-A8C3-9DCE37E6EC90"}`
- Canonical: `{"device_eui":"0016C001F11715E2","event_uuid":"d4fe4b8f-2c58-4d1c-a8c3-9dce37e6ec90"}`
- SHA-256: `5a6af3263b6ee1803f572b1011b72f2e1e5ba72e4c45f0ca46095fc6f477d8e5`

### Vector 7 — EUI-48 expansion
- Input: `{"device_eui": "0016c001f117"}`
- Canonical: `{"device_eui":"0016C0FFFE01F117"}`
- SHA-256: `7af6b7cc1190a447fddaf87749fdb4418938585b205738b20c32de051b173741`

### Vector 8 — UTF-16 object-key order
- Input: `{"\uE000": 2, "\uD800\uDC00": 1}`
- Canonical: `{"𐀀":1,"":2}`
- SHA-256: `4045c21a23c8ae8f8d9add81f54bd506bee65885099876fb4afb378b1f2c3516`

U+10000 sorts before U+E000 because its first UTF-16 code unit is U+D800.

> Test-vector hashes above were generated by the Java SyncPayloadCanonicalizer (osi-server). All runtimes lock to these values.

## Rejection vectors

The following input is normatively invalid. Canonicalization must fail without
producing a canonical body or hash.

### Timestamp fraction beyond nine digits

- Input: `{"note": "2026-05-03T12:00:00.1234567890Z"}`
- Result: invalid

## SWT pF Derivation

pF is never stored or synced for measurements; every consumer derives it from
canonical kPa with this exact rule:

- `pF = log10(kPa * 10)` (pF is the decimal logarithm of tension in hPa; 1 kPa = 10 hPa)
- `kPa = 10^pF / 10`
- `NULL`, non-finite, or `<= 0` kPa derives `null` (no clamping, no substitute values)
- Display rounding: pF `toFixed(2)`; CSV export value: 4 decimals
- CSV pairing: each SWT kPa row is paired with a row whose `channel_key` gains the
  `_pf` suffix, `unit` is `pF`, and `series_label` gains the ` (pF)` suffix

Golden vectors (all implementations - edge JS, GUI TS, server Java - must match):

| kPa | pF (exact double) | pF @ 2 dp | pF @ 4 dp |
|-----|-------------------|-----------|-----------|
| 10 | 2.0 | 2.00 | 2.0000 |
| 30 | 2.4771212547196626 | 2.48 | 2.4771 |
| 60 | 2.7781512503836436 | 2.78 | 2.7782 |
| 300 | 3.4771212547196626 | 3.48 | 3.4771 |
| 0 | null | null | null |
| -5 | null | null | null |
| null | null | null | null |

## Conformance

A new runtime must pass every test vector before its hashes are accepted by other systems. Hashes computed by a non-conformant runtime are discarded.

## Versioning

This document is `v1`. Breaking changes require a new file `canonicalization-v2.md` and a coordinated rollout with deprecation period.
