# Sync Payload Canonicalization

This document pins the canonical form used to compute payload hashes for sync events. All runtimes that compute or verify hashes must produce byte-identical output for the same logical payload.

## Algorithm

1. Project the event payload to its canonical body (transport metadata excluded — no batch wrapper, no auth context, no receipt timestamp).
2. Normalize per the rules below.
3. Serialize as compact JSON (no insignificant whitespace).
4. Encode UTF-8.
5. Hash with SHA-256.
6. Express as lowercase hex (64 chars).

## Rules

### Object keys
Sorted lexicographically (Unicode code point order). Stable across nested objects.

### Whitespace
None. No spaces between tokens. No trailing newlines.

### Strings
UTF-8. Standard JSON escape: `\"`, `\\`, `\b`, `\f`, `\n`, `\r`, `\t`, and `\uXXXX` for code points < 0x20. Do not use `\uXXXX` for printable ASCII or for code points >= 0x20 that JSON permits literally.

### Timestamps
ISO-8601 UTC with **fixed millisecond precision**: `YYYY-MM-DDTHH:MM:SS.sssZ`. Sub-millisecond precision is truncated (not rounded). Timezone is always `Z`; offset forms are converted to UTC at canonicalization time.

### Numbers
Fixed-point decimal string. No exponent form. No negative zero.
- Integers: bare digits, no trailing `.0`. Example: `42`, not `42.0`.
- Floats: up to 9 significant digits, trailing zeros trimmed except one digit after the decimal point. Example: `3.14`, `0.000001`, `1.5`.
- Never `NaN`, `+Infinity`, `-Infinity`; reject the payload at canonicalization time if encountered.

### Null vs absent
`null` is preserved. Absent fields stay absent — never serialized as `null`.

### UUIDs
Lowercase, hyphenated 8-4-4-4-12. Example: `d4fe4b8f-2c58-4d1c-a8c3-9dce37e6ec90`.

### EUIs
Uppercase, no separators. Example: `0016C001F11715E2`. EUI-48 input is expanded with the standard `FFFE` rule before canonicalization.

### Booleans
`true` and `false` lowercase, as standard JSON.

## Test Vectors

The following pairs are normative. A runtime is conformant if its hash matches the expected value for every input.

### Vector 1 — empty object
- Input: `{}`
- Canonical: `{}`
- SHA-256: `44136fa355b3678a1146ad16f7e8649e94fb4fc21fe77e8310c060f61caaff8a`

### Vector 2 — sorted keys
- Input: `{"b": 1, "a": 2}`
- Canonical: `{"a":2,"b":1}`
- SHA-256: `d3626ac30a87e6f7a6428233b3c68299976865fa5508e4267c5415c76af7a772`

### Vector 3 — timestamp precision
- Input: `{"recorded_at": "2026-05-03T12:00:00.123456789Z"}`
- Canonical: `{"recorded_at":"2026-05-03T12:00:00.123Z"}`
- SHA-256: `b2433460d6039d8b8bb640cb46d854e5fcdc6c511a4152ba0f9add660ceea9cc`

### Vector 4 — float fixed-point
- Input: `{"flow_rate_lpm": 8.5e0}`
- Canonical: `{"flow_rate_lpm":8.5}`
- SHA-256: `890beb02858df19a0d452b7a3bac58118a180b2e61d41e810d10768c18acb727`

### Vector 5 — null preserved, absent dropped
- Input: `{"a": null, "b": null}` with field `c` absent
- Canonical: `{"a":null,"b":null}`
- SHA-256: `052c4bd5e6ded53bd884485af8b1667a7b70ba3a8573b54bd878f6d2c705c2df`

### Vector 6 — EUI uppercase, UUID lowercase
- Input: `{"device_eui": "0016c001f11715e2", "event_uuid": "D4FE4B8F-2C58-4D1C-A8C3-9DCE37E6EC90"}`
- Canonical: `{"device_eui":"0016C001F11715E2","event_uuid":"d4fe4b8f-2c58-4d1c-a8c3-9dce37e6ec90"}`
- SHA-256: `5a6af3263b6ee1803f572b1011b72f2e1e5ba72e4c45f0ca46095fc6f477d8e5`

> Test-vector hashes above were generated by the Java SyncPayloadCanonicalizer (osi-server). All runtimes lock to these values.

## Conformance

A new runtime must pass every test vector before its hashes are accepted by other systems. Hashes computed by a non-conformant runtime are discarded.

## Versioning

This document is `v1`. Breaking changes require a new file `canonicalization-v2.md` and a coordinated rollout with deprecation period.
