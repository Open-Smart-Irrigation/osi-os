# Plan — `treated_area` optional everywhere + prefilled from plot area (catalog v8)

**Date:** 2026-07-22
**Branch:** design-sync/agrolink (worktree /home/phil/Repos/osi-os-agrolink)
**Maintainer decision (confirmed):** make `attr.treated_area` **optional for ALL activities** (including the dosing ones — fertilization/fertigation/plant-protection/seeding/planting), keep it **visible** where relevant, and **default its value from the plot's own area (`journal_plots.area_m2`)** so a routine full-plot entry shows it prefilled and editable. Rationale: selecting a plot that already has an area means you treat the full plot, so re-entering (or being gated on) the area is friction.

Follows the shipped W1/W2/W3 (`e5734796`) and the drift repair (`270db104`). `treated_area` was only relaxed for irrigation there indirectly; this makes it uniformly optional.

## Grounding (verified by investigation — re-verify in code)
- Plots carry `journal_plots.area_m2` (REAL, nullable; set in `PlotForm.tsx` "Area (m²)"). Can be null (a plot may have no area on file).
- `attr.treated_area` = `numberAttribute('attr.treated_area','Treated area','area','land_area','unit.m2_area',{min:0})` (`scripts/journal-catalog-core.js:142`). Unit is m² — same as `area_m2`, no conversion.
- Today `treated_area` is **required** via two paths:
  1. **`open_field` layout `minimum_fields`** (`['attr.block_bed_row','attr.treated_area','attr.cover_type','attr.denominator']`) → GUI-only force-require for non-quick templates (via the shipped `templateEngine` decouple, treated_area stays required because it is NOT in `static_context_fields`). Greenhouse/lysimeter use their own area fields (`plant_area`/`surface_area`), NOT `treated_area`, so only `open_field` carries it in minimum_fields.
  2. **`full_record.activity_requirements`** requires `attr.treated_area` for `fertilization`, `fertigation`, `plant_protection_application`, `seeding`, `planting_transplanting` (edge + GUI enforce this).
- **Visibility** of `treated_area` per activity comes from `full_record`'s `operation_fields_by_activity` (`FULL_RECORD_V6_OPERATION_FIELDS_BY_ACTIVITY`, reused by v7): it lists `treated_area` for fertilization/fertigation/plant_protection_application/seeding/weed_control_nonchemical/tillage_soil_work/mowing but **NOT for irrigation** (and the implementer must CHECK planting_transplanting).
- Edge validation enforces `activity_requirements`/`conditional_groups` only — never layout `minimum_fields` (`osi-journal/index.js`, `definition.js`). Nothing downstream computes anything from `treated_area` (grep: only templateEngine/carryForward/tests reference it; `denominator_contract` is declared-but-unused).
- Versions: `full_record`=v7, `open_field`=v3, global catalog=v7=migration `0029` (shipped). Generator sets a template/layout row's global `since` = its `version` field. **Next = v8 = migration `0030__journal_catalog_v8.sql`.**

## Task 3 — GUI: prefill treated_area from plot area (visible, editable) — Fable-tightened
Goal: when a plot with an area is selected, its `area_m2` shows up **prefilled in the visible `treated_area` input**, editable, overridable, and NOT stale after a plot switch. Fable flagged the naive approach fails — follow these specifics exactly:

1. **Seed the `values` state, NOT `payloadValues`.** The static-context prefill (`plotContextInputs`, `JournalCaptureFlow.tsx:486-512`) is merged into `payloadValues` at save-assembly (~`:1089-1094`) and renders read-only — copying that pattern for `treated_area` would RECORD the value but leave the input BLANK (fails "visible prefilled"). The editable field renderer is driven by `values` (seeded ~`:723`), so the area default must land in `values`.
2. **Inject on plot selection, not just at mount.** `values` is re-seeded on `selectPlot` (→ `withCanonicalContextCrop(...)`, ~`:1652-1664`, where `nextPlot.area_m2` is available ~`:1613`), on layout change (~`:1720`), and on activity pick (~`:1751-1755`, must survive `sanitizeValues`). The common flow starts at the `where` step with no plot, so a mount-only seed never fires. Inject the area default in `selectPlot` (and ensure it survives layout/activity transitions).
3. **Use the number-attribute value shape** (mirror `plotContextInputs` ~`:499-506`): `{ entered_value_num: area_m2, entered_unit_code: 'unit.m2_area' }` — NOT a plain string. Same unit as `area_m2` (m²), no conversion.
4. **User-edit-wins + re-apply-on-plot-switch via the existing `automaticPrefillRef` machinery.** Register the area default the same way carry-forward auto-prefills are owned, so `withoutUnchangedOwnedValues` (`:1373-1387`, invoked by `clearOwnedCarryForward` in `selectPlot`/`chooseLayout`/`pickActivity` ~`:1623,1709,1738`) (a) lets a user-typed `treated_area` win and (b) clears/re-applies the default when the plot changes — preventing a stale previous-plot area from carrying over. (If not routed through `automaticPrefillRef`, you MUST instead: seed only when the retained `values` has no user-edited `treated_area`, and recompute from the new plot's `area_m2` on every plot change — the `automaticPrefillRef` route is cleaner and preferred.)
5. If `area_m2` is null/absent → leave `treated_area` blank (optional). Do not fight a carry-forward/repeat-treatment value.

Read the real accessor/ref names in the current file before editing; the line numbers above are Fable's references, not guarantees.

## Risks / watch-items
- **Render-path completeness (highest — Fable-verified):** only `irrigation` needs `treated_area` added to operation_fields; the 5 dosing activities (incl. planting_transplanting via inheritance) + weed/tillage/mowing already carry it, so dropping them from `activity_requirements` does NOT hide the field. Confirm all 8 still show it post-change.
- **Intended visibility narrowing (not a regression — Fable flag):** `treated_area` is currently force-*required*+visible on ALL 16 activities in Full mode (via `open_field.minimum_fields`). Removing it from `minimum_fields` makes it DISAPPEAR entirely from the 7 activities that don't carry it in operation_fields: **pruning, crop_care, harvest, sampling, general_observation, pest_disease_observation, equipment_maintenance** (harvest keeps its own `harvest_area`). This is intended/an improvement (a required treated-area on a general observation is nonsense) — call it out in the live-verify and PR body so it isn't mistaken for a regression, and confirm nothing on `harvest` depended on `treated_area`.
- **PPP compliance NIT (PR-body line, Fable):** treated area is a standard field on plant-protection/spray records under EU SUD / Swiss ChemRRV / GLOBALG.A.P.; a cleared `treated_area` on a PPP entry is a small compliance gap, mitigated by the plot-area prefill in the common full-plot case. Acceptable per maintainer; note it, and if a future audit-export needs per-op area, re-require it for PPP then.
- **Version-pinned edge behavior:** old entries pinned to full_record@6/@7 still require treated_area on the dosing activities (their frozen activity_requirements). That's correct/consistent; note it so it isn't mistaken for a bug.
- **Prefill must merge UNDER user input** and not fight carry-forward/repeat-treatment prefills.
- **v8 layout bump:** `open_field` MUST be `version: 8` (global since=8 → migration 0030). A wrong tag (e.g. 4) would rewrite a frozen migration and abort generation (Slice-1 lesson).
- **Agronomy note for the PR:** treated_area is now optional even for dosing; acceptable because nothing computes rate from it today and the plot-area prefill still records it. If a future rate feature needs per-op area, re-require it then.
