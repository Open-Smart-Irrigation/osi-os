## 1. Executive Verdict

**Not ready.** The global-settings spec is *mostly ready* after trims (cut the history-range default, rewrite the dark-mode expectations, specify bulk-timezone semantics). The pF spec needs a **fundamental revision**: the "complete stored derived chain" should be dropped. pF is a pure, strictly monotonic function of kPa; storing it buys almost nothing on this system (every consumer is JS or Java) while pulling in the three heaviest change classes the repo has: a data backfill on the history-bearing tables, sync-trigger replacement, and a cross-repo contract change — all of which currently have **no sanctioned delivery path to live gateways** (boot node frozen, migration runner not on-device, `deploy.sh` has no migration hook). The pF-space scheduler comparison is also semantically wrong: because `log10` is concave, the same physical threshold authored in pF vs kPa triggers at different times.

Separately, the review found a **latent P0 bug** the pF spec builds on top of: the seed and bundled DB constrain `irrigation_schedules.trigger_metric` to `('SWT_WM1','SWT_WM2','SWT_AVG')` while the runtime API, GUI, and cloud all use `SWT_1/2/3/DENDRO` — a fresh-seeded gateway cannot save a DENDRO or SWT_1/2/3 schedule at all.

The good news: the revised design (authoritative unit + always-recomputed kPa mirror, pF derived at display/export time) delivers the entire user-visible feature — pF display, pF threshold authoring, pF CSV columns — with **two additive columns on one non-history table and zero scheduler/trigger/contract changes**.

## 3. Highest-Risk Gaps (prioritized)

### 3.1 P0 (pre-existing, newly confirmed): the schedules CHECK rejects the live metric vocabulary
`database/seed-blank.sql:152` and the bundled `farming.db` both carry
`trigger_metric TEXT NOT NULL CHECK (trigger_metric IN ('SWT_WM1','SWT_WM2','SWT_AVG'))`.
But the schedule-save flow (`Verify Zone Ownership`, node `22cc64fa2a899cea`) allows `['SWT_AVG','SWT_1','SWT_2','SWT_3','DENDRO']`, the GUI authors DENDRO schedules (`AdvancedScheduleDrawer.tsx`), the cloud contract enum includes DENDRO, and the cloud→edge `UPSERT_SCHEDULE` command writes whatever metric the server sends (`Build UPDATE SQL`, node `4f4a765f36cee6f3`). Neither the boot node (`sync-init-fn` — no `irrigation_schedules` CHECK handling) nor `scripts/repair-pi-schema.js` widens it. **Failure mode:** on any freshly seeded gateway, saving a DENDRO or SWT_1/2/3 schedule fails with a CHECK violation (HTTP 500 / silent FAILED sync ack); cloud-pushed schedules fail server-invisibly. This is the same defect class as the documented `devices` CHECK regression. It must be fixed (and is a table rebuild — the CHECK can't be ALTERed) before the pF spec's premise "the scheduler keeps the existing metric set SWT_1, SWT_2, SWT_3, SWT_AVG" is even true. Also note the spec's metric list omits DENDRO entirely.

### 3.2 The stored pF chain has no way to reach the fleet
- The `DEVICE_DATA_APPENDED` payload is built by an explicit `json_object(...)` inside `trg_dp_device_data_outbox_ai` (`seed-blank.sql:1275-1351`); adding `swt_*_pf` requires DROP+CREATE of the trigger → `destructive` class per AGENTS.md, and "trigger replacement" is an explicit **Option B promotion condition** in the ADR.
- Backfilling `device_data`/`chameleon_readings` is a `data`-class migration — also an Option B condition.
- The boot node is **frozen** for new schema; the Phase-1 runner "does not yet run on-device"; `deploy.sh` contains no migration invocation. There is literally no sanctioned mechanism today to add these columns to Silvan/kaba100/Uganda.
- Even if delivered: backfill UPDATEs emit no sync events (triggers fire on INSERT — AGENTS.md documents this exact pitfall), so the cloud mirror would permanently lack pF for all pre-change history unless osi-server runs its own backfill. "CSV always includes both units" then silently breaks for historical rows on any consumer reading mirrored values.

### 3.3 pF-space comparison changes irrigation behavior
The scheduler computes an **arithmetic hour-mean** of per-row channel means in kPa (`Build mean query`, node `d0b2b1c1a937e16d`: `AVG(expr)` over the last hour) and compares it to `threshold_kpa` (`Decide + build actuator cmd`, node `5f0d2b7e9b9b1b3a`), logging `aggregate_kpa`/`threshold_kpa` into `irrigation_events` (kPa-typed columns, also in the sync payload, and NOT NULL kPa on osi-server's `IrrigationEvent`). Averaging stored pF values instead is averaging logarithms: by Jensen's inequality `mean(pF) ≤ pF(mean kPa)`, so a schedule authored at pF 2.48 triggers later (drier) than the identical schedule authored at 30 kPa. Unit choice must never change trigger semantics. (Geometric-mean aggregation of tension is scientifically defensible — but if wanted, change it for *all* schedules as a deliberate, separate decision, not as a side effect of the authoring unit.)
Additional concrete hole: the scheduler expressions fall back to legacy channels (`COALESCE(dd.swt_1, dd.swt_wm1)`), but pF columns were only proposed for `swt_1..3` — pF-space evaluation would silently exclude legacy Kiwi rows (`swt_wm1/wm2`), and `SWT_WM1`/`SWT_WM2` metrics would have no pF source at all.

### 3.4 `threshold_kpa` is NOT NULL in both repos and semantically overloaded
Edge: `seed-blank.sql:153` `threshold_kpa REAL NOT NULL`. Cloud: `IrrigationSchedule.java:27-29` `nullable = false`. So "threshold_kpa **may** be populated" is impossible — the kPa mirror is structurally mandatory, which is good: make it the invariant. Worse, for DENDRO the field is not kPa at all but an encoded stress level 1–4 (edge validation in `Verify Zone Ownership`; decoding in `AdvancedScheduleDrawer.tsx:41` and osi-server `DendroAnalyticsService.dendroThresholdStressLevel`). `threshold_unit='pF'` against a DENDRO schedule is meaningless and must be rejected.

### 3.5 Round-trip clobber via cloud commands
`UPSERT_SCHEDULE` from pending-commands (node `4f4a765f36cee6f3`) upserts `trigger_metric, threshold_kpa, duration, enabled, response_mode` — nothing else. An old cloud editing a pF-authored schedule would silently reset/orphan `threshold_unit`/`threshold_pf`. Under the kPa-runtime model this only stales display intent (behavior stays correct); the spec must still define the rule (recommended: a unit-less command means kPa-authored ⇒ set `threshold_unit='kPa'`, recompute `threshold_pf`).

### 3.6 Existing Schedule contract drift
`docs/contracts/sync-schema/resources.schema.json:34-45` — Schedule enum is `["SWT_WM1","SWT_WM2","SWT_AVG","DENDRO"]` (missing `SWT_1/2/3` the runtime accepts) and names the field `trigger_value` while live payloads emit `threshold_kpa` (`seed-blank.sql:1260`). Layering `threshold_unit`/`threshold_pf` on a drifted contract widens the exact class the ADR was written to close. Fix the drift as its own commit before (or with) any schedule payload change.

### 3.7 Dark mode's CSS-variable premise is mostly aspirational
`:root` variables exist (`index.css:4-19`, ~20 vars) but `var(--…)` usage is ~35 references confined to `App.css`/`index.css`, while `.tsx` files contain **~194 hardcoded hex colors** in inline styles (cards, charts, statuses). Flipping root variables will restyle the shell and leave most card/chart interiors light — "visually broken" by the spec's own standard. `WindRoseChart.tsx:39` already observes `data-theme`, so the mechanism is right; the effort estimate is not.

### 3.8 Fleet delivery prerequisite (affects both specs' only schema change)
Even the minimal delta (2 additive columns on `irrigation_schedules`) has no live-Pi path: boot node frozen, runner not invoked anywhere at deploy. This feature is the first schema-touching product work since the freeze — the "runner invoked from `deploy.sh` (backup + `verifyHead` + fail-closed)" step must be designed and rehearsed first, or the feature ships fresh-flash-only with the GUI feature-detecting the columns. Decide explicitly.

## 4. Spec Patch Suggestions

### `2026-07-05-swt-pf-unit-support-design.md`
1. **Replace the Decision section** with: kPa remains the only stored measurement unit. pF is derived on demand by shared converters (`kpaToPf`/`pfToKpa`) in the React GUI, the Node-RED API/CSV layer, and osi-server. The only stored pF value is the schedule authoring intent: `irrigation_schedules.threshold_unit TEXT NOT NULL DEFAULT 'kPa' CHECK (threshold_unit IN ('kPa','pF'))` and `threshold_pf REAL`.
2. **Delete the Database section's** six `swt_*_pf` columns on `device_data` and `chameleon_readings`, and the historical backfill.
3. **Threshold authority invariants** (new wording):
   - `threshold_kpa` is always populated (it is NOT NULL in both repos) and is always `pfToKpa(threshold_pf)` recomputed server-side at save time when `threshold_unit='pF'`; client-supplied mirror values are ignored.
   - The scheduler always compares in kPa against `threshold_kpa`. `threshold_unit`/`threshold_pf` are authoring/display metadata; they never alter runtime evaluation.
   - `threshold_unit='pF'` is valid only for SWT metrics; DENDRO schedules are unit-less encoded levels — the API rejects `pF` + `DENDRO` with 400.
   - pF input validation: convert to kPa first, then apply the existing `0 < kPa ≤ 300` rule (single source of range truth; equivalently `pF ≤ 3.4771`).
   - Old edge clients / unit-less cloud `UPSERT_SCHEDULE` commands ⇒ `threshold_unit='kPa'`, `threshold_pf` recomputed.
4. **SWT_AVG section**: delete the pF-averaging branch. State: aggregation semantics are unit-independent (arithmetic hour-mean of kPa, unchanged); note the geometric-mean question as explicit future work requiring a global decision.
5. **API/Sync section**: edge API responses include `threshold_unit`, `threshold_pf`, and computed `swt_*_pf` beside kPa values (computed at response time). Sync payloads and `DEVICE_DATA_APPENDED` are **unchanged in this increment**; osi-server computes pF from kPa where it needs it. Adding unit fields to `SCHEDULE_UPSERTED` requires replacing `trg_sync_schedules_outbox_au` (including extending its change-detection `WHEN` clause, `seed-blank.sql:1239-1247`) — defer to the Option B window, and update `resources.schema.json` in the same change (also fixing the existing enum/`trigger_value` drift).
6. **CSV Export**: pF columns are computed at export time in the history router from kPa. Note the corollary: this makes "both units, always, for all history" trivially true — no backfill needed.
7. **Migration section**: reclassify honestly — one `additive` ordered migration (0002) for the two schedule columns; name the delivery mechanism for live Pis (deploy-time runner invocation) as a prerequisite; state that the seed CHECK fix (3.1) is a `destructive` rebuild of `irrigation_schedules` handled as its own migration.
8. **Precision**: store `threshold_pf` exactly as authored (full REAL); never round at storage; display pF at 2 dp; display of the authored threshold always reads the authoritative field, never the recomputed mirror.
9. Add DENDRO to the "existing trigger metric set" statement (factual fix) and mention the legacy `swt_wm1/wm2` fallback path.

### `2026-07-05-global-settings-page-design.md`
1. **Cut `Default history range`** from the first increment — it duplicates existing per-card server-side preferences (`PUT /api/history/zones/:zoneId/cards/:cardId/preferences`) and creates precedence confusion the spec already has to caveat.
2. **Timezone bulk apply — specify**: implemented as one ownership-scoped statement (`UPDATE irrigation_zones SET timezone=?, updated_at=?, sync_version=sync_version+1 WHERE user_id=? AND deleted_at IS NULL AND timezone <> ?`), atomic by construction; the existing `AFTER UPDATE` zone trigger (timezone is in its change-detection list, `seed-blank.sql:1083`) emits one `UPSERT_ZONE` per changed zone, so sync rides existing rails; response reports `updatedZones` (rows actually changed). Align `sync_version` handling with the existing per-zone timezone handler. Confirmation dialog must state: "affects scheduler timing, new daily rollups, and history range boundaries from now on; already-recorded daily history is not recomputed."

5. **Theme — rewrite the increment**: first shippable dark mode covers the shell (header, page background, card container surfaces, primary text) via the existing variables; chart/data-viz interiors are explicitly exempted (kept on light card surfaces) until the ~194 inline hex colors are swept component-by-component (WindRoseChart's `data-theme` observer is the pattern). The settings page ships independently; theme can land as its own slice.
6. **Endpoint naming**: consider `PUT /api/irrigation-zones/timezone` for the bulk action to stay in the existing route namespace instead of introducing `/api/settings/*`; either way, one namespace decision, stated.
7. Add to Non-Goals: no settings-driven change may alter `irrigation_events`/scheduler semantics (the SWT unit preference is display-only; the schedule editor's unit selector is the only semantic unit control).

## Addendum (2026-07-05, after Phil's answers)

**Phil's decisions:** CSV always carries both units by default (confirmed); shell-first dark mode accepted (Q5). Derive-at-edge vs derive-at-write sent to an independent senior-engineer review.

**Q2 resolved — the P0 CHECK bug is confirmed live and was silent by luck:**
- Both demo gateways verified over SSH: Silvan and kaba100 carry the narrow `CHECK (trigger_metric IN ('SWT_WM1','SWT_WM2','SWT_AVG'))` in their live `/data/db/farming.db`.
- Silvan has zero schedule rows ever. kaba100 has exactly one: `SWT_WM1, 30.0 kPa`, created 2026-04-16, now disabled.
- Timeline: `ef6688f2` (2026-06-24) widened only the flow API validation to accept `SWT_1/2/3`; `d59c2934` (2026-06-25) switched the GUI to canonical metrics — defaulting to `SWT_1` and normalizing legacy `SWT_WM1 → SWT_1` on edit. No commit ever widened the CHECK in `seed-blank.sql`, the bundled DBs, the boot node, or `repair-pi-schema.js`.
- Net effect since 2026-06-25: in the current GUI only "Mean (all sensors)" (`SWT_AVG`) can be saved. `SWT_1/2/3` and `DENDRO` saves hit the CHECK → sqlite node error → `device-api-catch` → HTTP 500 `{error: 'device-api failed'}`. Editing kaba100's legacy `SWT_WM1` schedule would resubmit as `SWT_1` and also 500.
- Uganda (production) not inspected; same one-line `sqlite3` check needed in a maintenance window.

**Q1 resolved — independent senior-engineer second opinion (derive-at-edge vs derive-at-write): verdict A, derive at read/export time. "Not close."**
Key reasoning (full transcript was a separate agent run):
1. **B's "single materialized truth" is unachievable here, so its core benefit is fake.** The outbox trigger enumerates columns explicitly; without the destructive trigger replacement (an Option B condition) the cloud must derive its own pF anyway — so storing yields *both* stored edge values *and* Java-derived cloud values: more re-derivation surfaces than A, plus a stored copy to police.
2. **B's delivery path doesn't exist; A's is one function.** Both the on-demand export and the persisted `/data/exports` rollup CSVs funnel through `rawZoneExportRows`/`aggregateZoneExportRows` in `osi-history-helper/index.js` — a single choke point for adding paired pF output. B needs the runner on-device + a data backfill over irreplaceable production history + a coordinated cloud backfill (backfill UPDATEs emit no sync events).
3. **CSV "both units for all history" is structurally free under A, structurally broken under B** (pre-backfill rows, backfill-window rows, and cloud rows all need fallback derivation anyway).
4. **Float determinism across JS/Java:** last-ulp differences can only flip a 2-dp display within ~1e-15 of a rounding boundary — practically impossible and physically meaningless vs sensor precision (±0.1 kPa ≈ ±0.004 pF at 10 kPa). Mitigation: pin the formula + rounding rule (e.g. round-half-up, 2 dp) in the sync-schema contract with shared golden vectors (meshes with the codec golden-vector work already planned). Storing does not remove this: the GUI still formats and the cloud still derives for un-synced rows.
5. **Pi debuggability** is A's only real cost (no SQL `log10` ad hoc); mitigated by decade landmarks (10 kPa = pF 2.0, 100 = 3.0), API/CSV emitting pF, and a `node -e` one-liner.
6. **Flip conditions to B (all required):** a validated need for per-sample-pF aggregates in raw SQL (JS rollups can already compute mean-of-pF if the geometric-mean semantics is ever adopted); runner live on-device with the trigger cutover done; a cloud backfill path.
