# Extract Get Zone Environment Summary → Pure Module + Thin Adapter

**Status:** Draft
**Refactor-program item:** 2.4 (second HTTP-shaped strangler seam; reuses the 2.2 "done" pattern)
**Focus: osi-os**
**Depends on:** 1.A1 (`osi-lib` loader — the mandated load mechanism defined in `2026-07-07-osi-lib-loader-design.md` §A/§B, with §E "Scope boundary" requiring every new seam to load this way, and §C giving the HTTP-shaped load-failure rule this node uses), 1.A2 (size ratchet — the merge gate proving the extraction lowered the scoreboard), and the pattern established by 2.2 (`2026-07-08-extract-daily-dendro-analytics-design.md` — same extraction contract, same "done", same golden-vector-capture-first structure; this spec mirrors it for an HTTP-request-path node).
**Feeds:** 4.2 (History API Router) reuses this HTTP-request-path extraction pattern; 2.4 is the second of two proofs 4.2's plan cites as its precedent.

## Problem

`Get Zone Environment Summary` (`flows.json` node id `zone-env-fn`, both profiles byte-identical) is a **67,317-char** function node — the 3rd-largest embedded-JS node in flows.json (measured 2026-07-08, `main` @ `98002778`). It is the **HTTP request-path** handler behind the zone environment/weather panel: on a GUI request it authenticates a bearer token, ensures its schema, reads local sensor climate + zone/gateway location + measured/estimated irrigation, optionally calls out to weather/forecast providers (OpenAgri, Open-Meteo), assembles an agronomic summary (VPD, THI, dew point, crop-coefficient ET, forecast, water balance), caches it, and returns the JSON bundle.

Unlike 2.2's node (a once-daily scheduler batch), this one runs **on a live request path** — which is exactly why its load-failure rule differs (osi-lib §C HTTP-shaped: `msg.statusCode = 503` + `module_unavailable`, never a dead node), and why the extraction's blast radius is a per-request 503 fallback rather than a missed daily tick.

The node interleaves five concerns in one body (verified by reading the node's 1425-line `func`):
1. **Auth + HTTP envelope:** `getAuthSecret`, `toBase64Url`, `fromBase64Url`, `verifyBearer`, `respond` — bearer verification and the response shaping. Impure (reads `env`, shapes `msg`).
2. **DDL:** `ensureSchema()` runs an idempotent migration array including 4 request-path `CREATE TABLE IF NOT EXISTS` statements (`zone_shared_environment`, `gateway_locations`, `zone_weather_cache`, `valve_actuation_expectations`) + `ALTER TABLE` + 2 `CREATE INDEX`. **HARD BOUNDARY — stays untouched (§ground-truth 3).**
3. **DB I/O:** `getCache`/`putCache` over `zone_weather_cache`, `getCachedSharedBundle`/`putSharedBundle`, `loadEstimatedIrrigationByLocalDate`, `buildWaterHistory`, `buildWaterEnvironment`, and the zone/device/location `SELECT`s in the main handler.
4. **Outbound HTTP:** `requestJson`, `buildFormBody`, `getLinkedServerTarget`, `fetchSharedBundle`, the OpenAgri (`fetchOpenAgriToken`/`Current`/`Forecast`) and Open-Meteo (`fetchOpenMeteoCurrent`/`Forecast`) provider calls, and the `resolveOnlineCurrent`/`resolveForecast`/`safeResolveOnlineCurrent`/`safeResolveForecast` orchestrators. Impure (`httpLib`/`httpsLib`).
5. **Pure compute/assembly** — ~40 math/stats/normalization/assembly functions (VPD, dew point, heat index, THI, metric extraction/aggregation, forecast merge + section building, agronomic summary, water-split overlay, display-status derivation, drift computation, ISO/timezone date helpers). **This is the extraction target.**

DD4's "done" bar (cite, don't re-derive — as re-scoped by 2.2 §C for I/O-heavy seams): **pure module + co-located `node --test` green in CI + golden vectors captured before extraction + loads via `osi-lib`; "adapter <~2 KB" re-read as "zero inline compute/business logic remains — the residual is auth + DDL + SQL I/O + HTTP orchestration only."** This node cannot hit a literal 2 KB (the auth/DDL/HTTP mass is irreducible), and 2.2 already recorded that DD4 refinement for I/O-heavy seams; 2.4 inherits it. What 2.4 delivers: extract the **pure compute/assembly core** (concern 5) into a tested `osi-zone-env` module loaded via `osi-lib`, shrinking the node's embedded JS and lowering the 1.A2 scoreboard, with the compute behavior pinned by golden vectors captured from the current node before the change.

## Verified ground truth

Measured/read directly from `conf/full_raspberrypi_bcm27xx_bcm2712/files/usr/share/flows.json` node `zone-env-fn` (1425 lines, 67,317 chars, libs `[osiDb, crypto, httpLib(http), httpsLib(https)]`) on `main` @ `f05b82ab` (origin/main; working tip `6322a07a`), 2026-07-08:

1. **The node is an HTTP request handler, NOT pure compute.** It authenticates, ensures schema, reads/writes SQLite, calls outbound weather HTTP, assembles a response, and caches it. **The module boundary this spec draws is: pure compute/assembly functions move out; auth + DDL + SQL + HTTP + `respond`/`node.*` orchestration stay in the adapter.**
2. **The extraction target (pure compute/assembly) is mostly context-free — with THREE wall-clock exceptions handled per §B.** Sampled `computeVPD`, `buildAgronomic`, `buildForecastSection`, `buildLocalEnvironment`, `resolveWaterAction`, `computeRecommendationDrift` reference no `node.*`, DB, `Date.now()`, or `env.get`. **BUT a full scan found three movers that read `new Date()`:** `parseOpenAgriForecast` (`observedAt: new Date().toISOString()`, unconditional), `mergeForecasts` (`|| new Date().toISOString()`, fallback), and `localDateIso` (`value ? new Date(value) : new Date()`, fallback when `value` is falsy). These are NOT pure as-is — they move only with the clock parameterized (an injected `observedAtMs`/`nowMs`, adapter supplies `Date.now()`), exactly as 2.2 parameterized `irrDecision`; otherwise they stay in the adapter. See §B. Every other §A mover is context-free. Each function is re-checked function-by-function during implementation (§B rule) — the plan's per-mover `new Date(`/`Date.now` grep is the guarantee; these three are the known non-pure cases and are pre-flagged.
3. **The `ensureSchema()` DDL block is load-bearing and stays in the adapter — HARD BOUNDARY.** It contains the 4 request-path `CREATE TABLE IF NOT EXISTS` strings (`zone_shared_environment`, `gateway_locations`, `zone_weather_cache`, `valve_actuation_expectations`) plus `ALTER TABLE` migrations and 2 `CREATE INDEX`. These embedded DDL strings are **the concern of Stage-2 / item 4.3 (boot-path DDL removal), NOT this item** — this extraction moves COMPUTE/ASSEMBLY logic out and must not alter, reorder, or relocate a single DDL statement. Two independent gates enforce this: (a) `verify-sync-flow.js:1696` REQUIRES `'CREATE TABLE IF NOT EXISTS zone_weather_cache'` to REMAIN in "Get Zone Environment Summary" (`expectIncludes`, verified) — moving it fails CI; (b) `verify-no-stray-ddl.js` is a must-not-increase ratchet on flows.json DDL markers — a pure module cannot own DDL, and moving DDL text out would drop the flows count while these functions are supposed to be compute-only. Behavior-preservation for the DDL is "byte-identical, untouched."
4. **HTTP weather integration stays adapter-side.** `requestJson`/`buildFormBody`/`fetchOpenAgri*`/`fetchOpenMeteo*`/`resolveOnlineCurrent`/`resolveForecast`/`safeResolve*` use `httpLib`/`httpsLib` (the node's imported `http`/`https` clients — `verify-sync-flow.js:1699` pins `const lib = urlString.startsWith('https:') ? httpsLib : httpLib;`). These are impure and remain in the adapter. In the golden-vector harness (§D) they are STUBBED deterministically so the extracted PURE assembly (which consumes their *results*, e.g. `mergeForecasts`, `buildForecastSection`, `parseOpenAgriForecast`) is exercised with fixed provider payloads.
5. **`buildAgronomic`/`buildForecastSection`/`mergeForecasts`/`buildLocalEnvironment`/`buildWaterEnvironment`-support functions are the highest-value pure targets** — they compute the farm-facing VPD/THI/ET/water-balance numbers the panel shows. Note `buildWaterEnvironment` itself is async + DB-reading (stays in adapter); the PURE pieces it composes (`mergeDailyIrrigationSplit`, `overlayLocalWaterIrrigationSplit`, `toEffectiveIrrigationMm`, `resolveWaterAction`, `buildSensorHealth`) move out and are individually vectored. The adapter's async `buildWaterEnvironment` keeps its SQL reads and calls the moved pure helpers.
6. **The precedent for "run the actual shipped function text" is `scripts/rehearse-devices-rebuild.js`** (verified — reads a node's `func` from flows.json and executes it against a real `node:sqlite` DB via `makeFacadeShim`). §D reuses this technique: it executes the CURRENT `zone-env-fn` body against fixture DB rows with stubbed HTTP + a fixed clock, and snapshots the returned environment bundle — those outputs ARE the golden vectors.
7. **The node's libs today are `[osiDb, crypto, httpLib, httpsLib]`.** Extraction adds `{"var":"osiLib","module":"osi-lib"}` alongside them; `crypto`/`httpLib`/`httpsLib` stay (auth + HTTP live in the adapter). Wiring to the node's HTTP-response chain is unchanged.
8. **HTTP-shaped load-failure rule (osi-lib §C, verified line 88).** On `!r.ok`, the adapter sets `msg.statusCode = 503; msg.payload = { error: 'module_unavailable', module: 'zone-env', message: r.error }; return [null, msg]` (matching the existing house 503 pattern) — the "503 quarantine per osi-lib §C" boundary the item names. Never a dead node on a live request path.

## Design

### A. Module boundary — what moves, what stays

**New module `conf/<profile>/files/usr/share/node-red/osi-zone-env/` (`index.js` + `package.json` + co-located `index.test.js`)** — pure Node, zero deps, registered in all three delivery surfaces per the osi-lib spec §E and gated by `verify-helper-registration.js` (added by 1.A1). Registered in `osi-lib`'s `NAME_TO_PATH` as `'zone-env': 'osi-zone-env'`.

**Moves into the module (pure compute/assembly core):** the context-free math/stats/normalization/assembly functions —
`toFiniteNumber`, `round`, `mean`, `median`, `minValue`, `maxValue`, `computeVPD`, `computeDewPoint`, `computeHeatIndexC`, `computeTHI`, `maxInstant`, `safeJsonParse`, `toIsoTime`, `cacheStatus`, `extractFirstMetric`, `extractMetrics`, `aggregateMetric`, `buildLocalEnvironment`, `resolveLocation`, `normalizeCloudServerUrl`, `normalizeSchedulingMode`, `normalizeDisplayMode`, `absoluteDelta`, `isIrrigationActionConflict`, `buildDisplayStatus`, `computeRecommendationDrift`, `bundleAgeMinutes`, `normalizePrecipitationProbability`, `findMetric`, `deriveCropCoefficient`, `estimateStepHours`, `sumRain`, `buildForecastSection`, `buildAgronomic`, `addUtcDays`, `toEffectiveIrrigationMm`, `buildSensorHealth`, `resolveWaterAction`, `mergeDailyIrrigationSplit`, `overlayLocalWaterIrrigationSplit`, plus the small string/tz normalizers `trimToNull`, `normalizeTimezone`, and the SQL-literal helper `s` **only if it is a pure string helper with no DB reference** (verify at implementation; if `s` is coupled to the adapter's query builder, leave it in the adapter). Each mover is confirmed context-free before it moves (§B).

**Moves WITH the clock parameterized (§B — the three not-quite-pure movers):** `parseOpenAgriForecast`, `mergeForecasts`, and `localDateIso` each read `new Date()` (a fresh wall-clock stamp) and so move only after gaining an injected `observedAtMs`/`nowMs` argument the adapter supplies as `Date.now()` — mirroring 2.2's `irrDecision` parameterization. These are the ONLY edited functions among the movers; all others are verbatim copies.

**Stays in the adapter (impure orchestration — the residual `func`):** the auth block (`getAuthSecret`, `toBase64Url`, `fromBase64Url`, `verifyBearer`), `respond`, the entire `ensureSchema()` DDL block (§ground-truth 3 — HARD BOUNDARY), all SQL I/O (`getCache`/`putCache`, `getCachedSharedBundle`/`putSharedBundle`, `loadEstimatedIrrigationByLocalDate`, `buildWaterHistory`, async `buildWaterEnvironment`, the zone/device/location `SELECT`s), the outbound-HTTP block (`requestJson`, `buildFormBody`, `getLinkedServerTarget`, `fetchSharedBundle`, `fetchOpenAgri*`, `fetchOpenMeteo*`, `resolveOnlineCurrent`/`resolveForecast`/`safeResolve*`, `unavailableOnlineSection`/`unavailableForecastSection`), and all `node.status`/`node.error`/`msg` shaping + the wall-clock capture. The adapter authenticates, ensures schema, reads rows, calls providers, hands the collected inputs to the module's pure functions, and shapes the response — the thin-adapter-over-pure-core shape.

### B. Handling not-quite-pure functions — per-function rule

Applied per-function during implementation and pinned by the plan:
- **A function moves only if it is context-free** (no `node.*`, no DB handle, no `httpLib`/`httpsLib`, no `env.get`, no `Date.now()`/wall-clock, no outer-scope mutation of adapter state). The §A list is the verified-context-free set at spec time; the plan re-confirms each with a grep of the extracted body before finalizing the move.
- **Async provider/DB functions do NOT move** — `resolveOnlineCurrent`, `resolveForecast`, `safeResolve*`, `buildWaterEnvironment`, `buildWaterHistory`, `getCache`/`putCache`, the `fetch*` family stay in the adapter; they are I/O orchestration. Their **pure sub-computations** (`mergeForecasts`, `buildForecastSection`, `parseOpenAgriForecast`, `buildAgronomic`, the water-split helpers) move out and the adapter functions call `ZE.<name>(...)`.
- **The three time-reading movers `parseOpenAgriForecast`, `mergeForecasts`, `localDateIso` move only with the clock parameterized** (verified — each calls `new Date()`): each gains an injected `observedAtMs`/`nowMs` argument; the adapter supplies `Date.now()`. `parseOpenAgriForecast` reads it unconditionally (`observedAt: new Date(observedAtMs).toISOString()`); `mergeForecasts` and `localDateIso` use it as the fallback where they currently call `new Date()` with no argument (`localDateIso(value, tz, nowMs)` → `const date = value ? new Date(value) : new Date(nowMs)`). This is behavior-preserving: the adapter passes the same instant it would have stamped. These three are the ONLY edited movers, mirroring 2.2's `irrDecision`. Any OTHER function found (during the plan's per-mover `new Date(`/`Date.now` grep) to read the wall clock gets the same treatment or stays — no function is smuggled into the pure module with a hidden ambient dependency.

### C. The adapter size reality — honest against DD4's re-scoped bar

Per 2.2 §C (already recorded as a DD4 refinement for I/O-heavy seams), the residual `zone-env-fn` cannot be 2 KB: the auth block, the `ensureSchema` DDL (~several KB), the multi-provider HTTP integration, and the SQL I/O are irreducibly large. What the extraction commits to:
- **The pure compute/assembly core (~40 functions in §A) leaves the node**, moving into `osi-zone-env/index.js`. The residual `func` is auth + DDL + SQL I/O + HTTP orchestration + `osiLib.require('zone-env')` + calls into the module. The **1.A2 total-JS scoreboard drops** by the moved char mass (module code lives in a helper file, not flows.json).
- **The re-scoped DD4 bar applies:** "the extracted pure core is a standalone tested module; the residual node is auth + DDL + I/O + HTTP orchestration only, with zero compute/assembly logic left inline." This is the honest bar for an HTTP-request-path node; 2.4 does not re-open the DD4 refinement (2.2 recorded it) but explicitly inherits it.
- The residual node still loads via `osiLib.require` (DD2/DD4), still gets its compute core proven by golden vectors, still lowers the scoreboard. The byte count it cannot meet literally is structural, not a shortcut.

### D. Golden vectors — captured from the CURRENT node before extraction (behavior-preservation LAW)

Golden vectors are captured by executing the **CURRENT** `zone-env-fn` body against fixture DB rows + stubbed HTTP + a fixed clock, using the `rehearse-devices-rebuild.js` technique (§ground-truth 6). The captured input/output pairs are the contract the extracted module must reproduce exactly.

**Two vector layers, both captured before the extraction touches the node:**

1. **Unit vectors for the pure functions (the module's own `node --test`).** For each function in §A, capture representative `(inputs) → output` pairs. Because they are self-contained, they are captured by extracting the current function source and running it against hand-built and real-data-derived inputs — e.g. `computeVPD(tempC, rh)`, `computeTHI(tempC, rh)`, `buildAgronomic(<local+online+forecast inputs>)`, `mergeForecasts(openAgri, openMeteo, { nowMs })`, `buildForecastSection(<merged forecast>)`, `parseOpenAgriForecast(<raw provider json>, { observedAtMs })`, `mergeDailyIrrigationSplit(<measured, estimated>)`, `overlayLocalWaterIrrigationSplit(<shared, local>)`, `computeRecommendationDrift(<a, b>)`, `resolveWaterAction(<inputs>)`, `localDateIso(<value>, tz, nowMs)`. These become `osi-zone-env/index.test.js` assertions — the module reproduces each output byte-for-byte. **The three clock-parameterized movers (`parseOpenAgriForecast`, `mergeForecasts`, `localDateIso`) are vectored with a FIXED `observedAtMs`/`nowMs` so their unit vectors are deterministic** (without the parameter their output would embed a fresh timestamp per call — the reason §B parameterizes them). **`computeVPD`/`computeTHI`/`computeDewPoint`/`deriveCropCoefficient`/`buildAgronomic` are farm-facing agronomy numbers — vectored with the osi-agronomy-sensors-reference formulas in mind and flagged for stronger-model review (Open decisions).**
2. **End-to-end golden vectors (the extraction proof).** A capture harness (`scripts/capture-zone-env-vectors.js`, kept in `scripts/`, not shipped to the Pi) seeds a `node:sqlite` DB from `database/seed-blank.sql` plus fixture rows for 2–3 zones (one with local sensors + explicit lat/long, one relying on the `gateway_locations` fallback, one with measured + STREGA-estimated irrigation), STUBS the weather HTTP deterministically (fixed OpenAgri/Open-Meteo payloads), pins the clock, executes the **current** node `func` against it via `makeFacadeShim`, and snapshots the returned environment bundle (the response `msg.payload`) into `docs/contracts/zone-env/` fixtures. Then the **extracted** node is run against the identical seed + stubs + clock and its output bundle is asserted deep-equal to the snapshot. This proves the whole request handler, post-extraction, produces an identical response — the strongest behavior-preservation evidence without a live gateway. HTTP is stubbed because the provider integration is adapter-side and out of the compute contract (Non-goals); the stub returns fixed payloads so the assembly is deterministic.

**Fixture location:** per-function unit vectors live beside the module (`osi-zone-env/index.test.js` / a `fixtures/` subdir); the end-to-end input→expected-bundle fixtures live in `docs/contracts/zone-env/` (this spec CREATES that directory + its first fixtures; a future cross-repo item may mirror them, mirroring 2.3's relationship to `docs/contracts/dendro/` — but 2.4 does not itself define a cross-repo gate).

### E. Load mechanism (cite osi-lib spec §A/§B + §C HTTP rule + §E, do not re-derive)

The module loads via `osiLib.require('zone-env')` per `2026-07-07-osi-lib-loader-design.md` §A/§B (the `osiLib.require` mechanism + `NAME_TO_PATH`) under §E's rule that every new seam MUST load this way: registered in `osi-lib`'s `NAME_TO_PATH`, packaged through the three delivery surfaces (runtime `package.json` + `package-lock.json`, `98_osi_node_red_seed` module loop, `deploy.sh` fetch pair), gated by `verify-helper-registration.js`. **The adapter's load-failure path follows osi-lib §C's HTTP-shaped rule (verified line 88):** on `!r.ok`, `msg.statusCode = 503; msg.payload = { error: 'module_unavailable', module: 'zone-env', message: r.error }; return [null, msg]` — never a dead node on the request path. `osiLib` is added to the node's `libs` alongside the existing `osiDb`/`crypto`/`httpLib`/`httpsLib`. `osi-zone-env` becomes a new non-codec `NAME_TO_PATH` entry, so `verify-helper-registration.js` enforces its three-surface registration at merge time.

## Non-goals

- **Changing any compute/assembly behavior.** Behavior-preserving extraction only; a captured golden vector that reveals a latent bug is documented, not fixed here (separate later PR per the program's Risks section).
- **Touching the `ensureSchema()` DDL block / the 4 request-path CREATE-TABLE strings** (§ground-truth 3, HARD BOUNDARY) — not moved, not reordered, not altered; those embedded DDL strings are Stage-2 / item 4.3's concern. `verify-sync-flow.js:1696` (`zone_weather_cache` must remain) and `verify-no-stray-ddl.js` (must-not-increase ratchet) both stay green.
- **The weather/forecast HTTP integration** (OpenAgri, Open-Meteo, shared-bundle fetch) — stays adapter-side, stubbed in the vector harness; not part of the compute contract.
- **The auth block** (`verifyBearer`/JWT) — stays adapter-side; not a pure-compute concern.
- **Any cross-repo contract gate** — 2.4 creates `docs/contracts/zone-env/` fixtures for behavior-preservation but defines no osi-server mirror (unlike 2.3 for dendro).
- **Deploying to any gateway.** No SSH, no live system.
- **Hitting a literal 2 KB adapter** (§C) — structurally impossible for an HTTP-request-path node; re-scoped per 2.2 to "zero inline compute/assembly logic remains."

## Definition of Done

- `osi-zone-env/` module (`index.js` pure, zero deps, + `package.json`) in **both** profiles, byte-identical, exporting the §A pure functions; registered in all three delivery surfaces + `osi-lib` `NAME_TO_PATH` (`'zone-env'`), green under `verify-helper-registration.js`.
- The §A functions are **removed from** `zone-env-fn`'s inline body; the node calls them via `osiLib.require('zone-env').value.*`; `osiLib` added to the node's `libs`; the `ensureSchema()` DDL block, auth block, SQL I/O, and HTTP orchestration unchanged.
- The HTTP-shaped 503 load-failure path (§E, osi-lib §C) is wired: `!r.ok` → `msg.statusCode = 503` + `module_unavailable`.
- `osi-zone-env/index.test.js` (`node --test`): per-function unit vectors (§D-1) for every exported function, green in CI.
- `docs/contracts/zone-env/` created with end-to-end input-rows → expected-response-bundle fixtures (§D-2); `scripts/capture-zone-env-vectors.js` captures them from the pre-extraction node and an assertion proves the post-extraction node reproduces them exactly (via the `node:sqlite` + facade-shim + stubbed-HTTP + fixed-clock harness).
- Both profiles byte-parity for every changed file (`verify-profile-parity.js` green); `verify-sync-flow.js` green (incl. the `zone_weather_cache` DDL and HTTP-client `expectIncludes` at lines 1696/1699); `verify-no-stray-ddl.js` not increased (DDL untouched); `verify-flows-size-ratchet.js` (1.A2) shows the flows.json total **decreased**.
- Frozen `sync-init-fn` untouched.
- CI-wired: the module test + the end-to-end capture/replay assertion run in `migrations.yml`.
