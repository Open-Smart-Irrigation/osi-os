# AgroLink Device Assignment and Configuration Parity Design

**Status:** Accepted for autonomous implementation

**Scope:** Task 8 row 3 of the AgroLink edge/cloud parity program

## Goal

Make portable device assignment, metadata, flags, and supported configuration
behave consistently on OSI OS and OSI Server while preserving edge authority
and the existing six-device product catalog.

This design does not add a device plugin system, change ChirpStack
provisioning, or make hardware observations into cloud-owned state.

## Current behavior

The edge `devices` row is already the canonical aggregate for:

- farm ownership and the primary irrigation-zone assignment;
- name and supported device type;
- dendrometer, temperature, rain-gauge, flow-meter, reference-tree, and
  Chameleon flags;
- Kiwi/Clover soil-sensor depths;
- Chameleon channel depths;
- the selected STREGA model; and
- the device aggregate `sync_version`.

The edge update trigger emits `DEVICE_ASSIGNED`, `DEVICE_UNASSIGNED`,
`DEVICE_FLAGS_UPDATED`, or `DEVICE_UNCLAIMED` with the complete portable
aggregate. OSI Server already mirrors most of those fields.

Cloud mutations do not yet follow one authority model. Assignment and unclaim
queue commands without a durable desired-state overlay. Flag and depth
endpoints first write the server mirror and then queue a command. The cloud
therefore presents unconfirmed state as canonical and can advance its version
before the edge accepts the edit.

SenseCAP S2120 multi-zone assignments use `weather_station_zones`. The edge
and server both expose the workflow, but the server currently changes its
local junction table without sending the change to the edge.

The LSN50, Kiwi, and STREGA hardware-setting endpoints already have matching
edge and cloud routes. These commands affect device firmware or radio
configuration. They are not canonical `devices` fields except for the LSN50
mode observation and STREGA model mirror.

The cloud dashboard has no registered section for `TEKTELIC_CLOVER` or
`AQUASCOPE_LORAIN`. An unassigned device of either type is omitted because
unassigned rendering iterates the static section registry.

## Product boundary

### Portable canonical state

The following fields belong to a protected device aggregate:

| Field | Rule |
|---|---|
| `device_eui` | Uppercase EUI-64; immutable resource key |
| `name` | Portable farmer label |
| `type` | One of the six supported families; immutable through this command |
| `claimed_user_uuid` | Edge-local user UUID mirrored to the selected gateway account |
| `zone_uuid` | Primary zone assignment, nullable |
| `dendro_enabled` | Valid only for `DRAGINO_LSN50` |
| `temp_enabled` | Valid only for `DRAGINO_LSN50` |
| `rain_gauge_enabled` | Valid only for `DRAGINO_LSN50` |
| `flow_meter_enabled` | Valid only for `DRAGINO_LSN50` |
| `is_reference_tree` | Valid only for a dendrometer-enabled LSN50 |
| `chameleon_enabled` | Valid only for `DRAGINO_LSN50` |
| `soil_moisture_probe_depths_json` | Valid only for Kiwi/Clover; positive integer centimetres |
| `soil_moisture_probe_depths_configured` | Travels with the depth map |
| `chameleon_swt1_depth_cm` through `chameleon_swt3_depth_cm` | Valid only for Chameleon-enabled LSN50; positive finite centimetres or null |
| `strega_model` | `STANDARD`, `MOTORIZED`, or null for STREGA only |
| `sync_version` | Exact edge-confirmed version |
| `deleted_at` | Existing tombstone semantics |

`current_state`, `target_state`, last-seen timestamps, decoder observations,
ChirpStack IDs, device mode observations, dendrometer ratios, baselines, and
calibration signatures do not enter desired state.

### Portable multi-zone state

S2120 weather-station assignments are a separate aggregate keyed by device
EUI. Its desired payload is a sorted, unique list of zone UUIDs plus an
independent version. It does not reuse the primary `devices.irrigation_zone_id`
version.

### Hardware configuration actions

These existing command families remain device actions:

- LSN50 mode, uplink interval, interrupt mode, and 5 V warm-up;
- Kiwi uplink interval and temperature/humidity enablement; and
- STREGA reporting interval, model, magnet mode, partial opening, flushing,
  and timed action.

Configuration actions remain durable; timed valve actions retain their short
expiry and physical-effect idempotency. The cloud must show their command
status but must not overlay the requested value onto an edge observation.
Payload hex and FPort are implementation details generated by the endpoint,
not user-controlled state.

The STREGA model is also a canonical metadata field. The model command may
update the edge row only after the edge accepts the hardware request; its
returning device event confirms the mirror.

### Edge-local workflows

The following stay edge-local:

- ChirpStack registration, application/profile selection, OTAA keys, and
  bootstrap;
- Chameleon calibration lookup and refresh;
- dendrometer ratio calibration, baseline capture/reset, and force-legacy
  diagnostics;
- raw firmware observations and decoder diagnostics; and
- physical gateway/device reachability checks.

The cloud may display mirrored diagnostic state read-only. It does not offer
an edit that pretends to perform a local calibration.

## Desired-state model

### Device aggregate

A capable gateway advertises `device_desired_state_v1`. Cloud edits use:

```text
UPSERT_DEVICE
device:<device_eui>:<base_sync_version>
```

The command carries the complete portable aggregate and requires
`target_sync_version = base_sync_version + 1`.

Unclaim uses:

```text
UNCLAIM_DEVICE
device_unclaim:<device_eui>:<base_sync_version>
```

It carries the same complete aggregate with `claimed_user_uuid = null` and
`zone_uuid = null`. Keeping unclaim in a separate effect family prevents a
safe metadata edit from superseding an ownership-removal operation.

One pending non-unclaim device edit may replace another unleased edit at the
same base. Unclaim is never coalesced with a metadata edit.

The server does not mutate `Device` when it creates either operation.
`DeviceResponse` overlays the desired aggregate and includes the operation
state. ACK plus a returning device event settle the operation. Equal-version
different-payload observations become a recoverable conflict.

Gateways without `device_desired_state_v1` keep the current assignment,
unassignment, unclaim, flag, and depth command paths. The server must not
pre-write their canonical mirror.

### Weather-station assignments

A capable gateway advertises `weather_station_zones_desired_state_v1`.
Cloud edits use:

```text
REPLACE_WEATHER_STATION_ZONES
weather_station_zones:<device_eui>:<base_sync_version>
```

The edge replaces the assignment set transactionally, increments its
independent version once, and emits `WEATHER_STATION_ZONES_REPLACED`.
The event and command use zone UUIDs, never repository-local numeric IDs.

Legacy gateways keep the current read-only cloud mirror. The cloud returns a
capability error instead of changing its own junction table.

## Authorization

Every mutation resolves the selected gateway membership and local user UUID.
The caller needs mutation permission for the device and every target zone.
A disabled membership is denied immediately.

Assignment rejects:

- a device or zone from another gateway;
- an unsupported or tombstoned device;
- a missing or tombstoned zone;
- a primary assignment to more than one zone; and
- a weather-station assignment containing an inaccessible zone.

The cloud user's global role grants no gateway authority.

## Supported catalog and UI

The advertised catalog remains:

- `KIWI_SENSOR`
- `TEKTELIC_CLOVER`
- `DRAGINO_LSN50`
- `SENSECAP_S2120`
- `AQUASCOPE_LORAIN`
- `STREGA_VALVE`

`MILESIGHT_UC512` remains accepted in stored schemas for compatibility but is
not shown as an AgroLink-supported device.

Cloud device rendering must register Clover and LoRain explicitly. Clover
reuses the Kiwi card only for fields actually populated by the mirror. LoRain
uses a rain-gauge card and never inherits soil-sensor settings. Unsupported
configuration controls are absent, not disabled placeholders.

Pending aggregate edits remain visible on the affected device card. Pending
primary assignment keeps the device in the desired zone with a status notice;
conflict or rejection exposes retry without hiding the canonical location.
S2120 desired assignments follow the same rule.

## Rollout

1. Add strict resource schemas, protected edge appliers, and tests while the
   new operations remain staged.
2. Advertise capabilities only after local route and command consumers pass.
3. Persist capabilities and accept new events on the server.
4. Switch cloud mutations to durable desired state and add UI overlays.
5. Add the S2120 assignment resource and the six-family rendering fix.
6. Vendor and activate the contract only after both consumers pass.

Legacy commands remain accepted during rollout. No existing gateway receives
a new command until its advertised capability is stored on the server.

## Verification

Completion requires:

- local-to-cloud and cloud-to-edge convergence for assign, unassign, flags,
  depths, STREGA model metadata, and unclaim;
- S2120 multi-zone convergence in both directions;
- replay, exact-version conflict, equal-version payload conflict, wrong
  gateway, disabled membership, and unauthorized-zone tests;
- legacy fallback tests;
- all six supported device families rendered in the cloud;
- device-family control visibility tests;
- hardware command expiry and effect-key tests;
- edge schema, flow, contract, profile, and scoped-access gates;
- server focused and full backend suites;
- frontend typecheck, focused tests, full tests, and build; and
- byte-identical server contract vendors.
