**Status:** Draft
**Refactor-program item:** 4.2 (the last and largest HTTP-shaped strangler seam — "the HTTP-shaped monster" DD4 names)
**Focus: osi-os**
**Depends on:** 1.A1 (`osi-lib` loader — `2026-07-07-osi-lib-loader-design.md` §A/§B mechanism + §C HTTP-shaped 503 rule + §E three-surface registration), 1.A2 (size ratchet — the merge gate proving the scoreboard dropped), 1.A3 (`osi-history-helper` co-located `node --test` — the callee this router leans on is under test before we extract the router around it), **and the extraction pattern proven TWICE by 2.2 (`osi-dendro-analytics`, scheduler-shaped) and 2.4 (`osi-zone-env`, HTTP-request-shaped).** DD4 explicitly orders this node LAST: "prove the pattern twice before the HTTP-shaped monster." ## Problem

`History API Router` (`flows.json` node id `history-api-router-fn`, both profiles byte-identical) is a **76,225-char** function node — the **largest** embedded-JS node in flows.json (measured 2026-07-08, `main` @ `f05b82ab`). It is the multi-route HTTP handler behind the history/analysis dashboard: on a GUI request it authenticates a bearer token, ensures its schema, parses the route + query params, resolves the owned zone/gateway context, reads history rows, delegates the heavy per-channel compute to the **already-extracted `osi-history-helper`** (it calls 16 distinct `osiHistory.*` methods across 31 call sites), assembles per-card series/summaries/calendars/interpretations, handles workspace + preference CRUD, and returns/streams the JSON (or CSV) response.

**Critical framing — 4.2 does NOT re-extract the helper.** `osi-history-helper` is already a tested module (1.A3); this router is its *main callee*. 4.2 extracts the **router's own pure glue** — the param-parsing, validation, range/season resolution, series-shaping, and response-formatting helpers that live inline in the router and are NOT in the helper — into a tested module, leaving auth + DDL + SQL I/O + `osiHistory.*` delegation + HTTP orchestration in the residual node. **A big-bang rewrite of this node is FORBIDDEN** (the program's Risks section names this node by size); the extraction is the same behavior-preserving, golden-vectored, one-function-family-at-a-time move that 2.2 and 2.4 established.

The node interleaves the same five concerns as 2.4's node (verified by reading its 1681-line `func`, 90 function declarations):
1. **Auth + HTTP envelope:** `getAuthSecret`, `toBase64Url`, `fromBase64Url`, `verifyBearer`, `respond`, `respondCsv`, `httpError`. Impure (reads `env`, shapes `msg`/streams).
2. **DDL:** `ensureHistoryTables(run)` runs 3 request-path `CREATE TABLE IF NOT EXISTS` (`zone_seasons`, `history_card_preferences`, `history_workspaces`) + indexes + a seed `INSERT`. **HARD BOUNDARY — stays untouched (§ground-truth 3).**
3. **DB I/O + helper delegation:** the `async` `getOwnedZoneContext`, `getGatewayContext`, `getActiveZoneSeason`, `getLatestDeviceDataRows`, `getLatestChameleonRows`, `getPreferenceRowsForScope`, `getGatewayLocationRow`, `getIrrigationEventSummary`, the workspace/preference CRUD (`listWorkspaces`, `createWorkspace`, `updateWorkspace`, `deleteWorkspace`, `upsertPreference`, …), and the `build*CardData`/`buildCardSummaries`/`buildAdvancedPayload` orchestrators that read rows and call `osiHistory.*`. Impure (DB + helper).
4. **Response streaming.** CSV export via `osiHistory.buildZoneExportCsv` + `respondCsv`.
5. **Pure router glue** — ~30+ param-parsing / validation / range-resolution / series-shaping / formatting functions (`parseZoneId`, `parseRangeSelection`, `validateView`, `validateAggregation`, `boolValue`, `numberOrNull`, `parseJsonObject`, `seasonRangeForContext`, `supportedRangesForCard`, `buildSeriesFromAggregate`, `truncateSeries`, `pointQuality`, `buildPreferenceMap`, `normalizeWorkspaceRow`, the source-classification predicates, the calendar/interpretation shapers, `safeFilenamePart`, …). **This is the extraction target.**

DD4's "done" bar (cite, don't re-derive — as re-scoped by 2.2 §C for I/O-heavy seams, inherited by 2.4): **pure module + co-located `node --test` green in CI + golden vectors captured before extraction + loads via `osi-lib`; "adapter <~2 KB" re-read as "zero inline compute/glue business logic remains — the residual is auth + DDL + SQL I/O + helper-delegation + HTTP orchestration only."** This node — the largest of the three — cannot hit a literal 2 KB, and 2.2 already recorded that DD4 refinement; 4.2 inherits it. What 4.2 delivers: extract the **pure router-glue core** (concern 5) into a tested `osi-history-router` module loaded via `osi-lib`, shrinking the node's embedded JS and lowering the 1.A2 scoreboard, with behavior pinned by golden vectors captured from the current node before the change.

## Design

### A. Module boundary — what moves, what stays

**New module `conf/<profile>/files/usr/share/node-red/osi-history-router/` (`index.js` + `package.json` + co-located `index.test.js`)** — pure Node, zero deps, registered in all three delivery surfaces per the osi-lib spec §E and gated by `verify-helper-registration.js`. Registered in `osi-lib`'s `NAME_TO_PATH` as `'history-router': 'osi-history-router'`. **This module is DISTINCT from `osi-history-helper`** (the helper does per-channel compute; this module does the router's request-glue). The router will load BOTH: `osiHistory` (the helper, unchanged) and the new `osiLib.require('history-router')`.

**Moves into the module (pure router-glue core) — the VERIFIED context-free set** (full-body scan of every candidate, 2026-07-08):
`safeFilenamePart`, `httpError`, `parseZoneId`, `boolValue`, `numberOrNull`, `parseJsonObject`, `sortIsoDesc`, `latestIso`, `supportedRangesForCard`, `seasonBoundaryIso` (its `new Date(parsed)` parses the INPUT `value`, not the wall clock — deterministic, pure), `seasonRangeForContext`, `parseRangeSelection` (with `nowMs` parameterized — §B; it is the ONE genuine wall-clock reader among the movers), `validateView`, `validateAggregation`, `isSoilSource`, `isEnvironmentSource`, `isIrrigationSource`, `isDendroSource`, `sourceKeyForDevice`, `pointQuality`, `soilChannelDepths`, `seriesWithDepth`, `buildSeriesFromAggregate`, `truncateSeries`, `latestPointTimestamp`, `latestValueBySeries`, `latestBatteryMetric`, `lastOpenedRankMap`, `buildPreferenceMap`, `normalizeWorkspaceRow`, `summaryScore`, `shouldUseHistoryRollups`, `rowHasSoilProfileValue`, `latestSeriesPoint`, `pointValueForCalendar`, `calendarRowsFromSeries`, `latestCalendarState`, `advancedField`, `knownAvailableFields`, `phaseSummary` (pure — takes its `phases` arg), and the display helpers `displayDeviceName`, `displaySourceDevices`, `displaySourceLabels` (verified: no `osiHistory`/`node.`/`crypto` reference; none duplicate an `osi-history-helper` export). Each mover was confirmed context-free AND non-duplicative-of-the-helper (the helper's 38 exports were diffed against this list — no collision).

**Stays in the adapter — HELPER-BOUND functions that call `osiHistory.*` (same class as `statusForCardValue`, §B):** `normalizeGatewayEui`, `uniqueDeveuis`, `displaySafeSourceKey` (all call `osiHistory.normalizeDeveui`), `soilRowsHaveWarning` (`osiHistory.classifySoilStatus`), `dendroRowsHaveWarning` (`osiHistory.classifyDendroStatus`), `environmentRowsHaveWarning` (`osiHistory.classifyEnvironmentStatus`). These are NOT moved — moving them would couple the glue module to `osi-history-helper` (the module has NO helper dependency, §E). They stay adapter-local.

**Stays in the adapter — WALL-CLOCK / ADAPTER-STATE helpers:** `nowIso` (it IS the clock — `new Date().toISOString()`, adapter-facing), `markPhase` / `addPhase` (both call `Date.now()` AND mutate the adapter-scope `const historyPhases` object — closure over adapter state, verified). These stay; only `phaseSummary(phases)` (pure, arg-driven) moves.

**Stays in the adapter (impure orchestration — the residual `func`):** the auth block (`getAuthSecret`, `verifyBearer`, `toBase64Url`, `fromBase64Url`), `respond`/`respondCsv`, the entire `ensureHistoryTables(run)` DDL block (§ground-truth 3 — HARD BOUNDARY), all SQL I/O and helper-delegation (`getOwnedZoneContext`, `getGatewayContext`, `getActiveZoneSeason`, `getLatest*Rows`, the workspace/preference CRUD, `build*CardData`/`buildCardSummaries`/`buildAdvancedPayload` and their `osiHistory.*` calls), the CSV export/stream path (`osiHistory.buildZoneExportCsv` + `respondCsv` — §ground-truth 4), and all `node.status`/`node.error`/`msg` shaping + the route dispatch. The adapter authenticates, ensures schema, reads rows, delegates compute to `osiHistory`, uses the new module's pure glue to parse/validate/shape, and returns the response — the thin-adapter-over-pure-glue shape.

### B. Handling not-quite-pure functions — the load-bearing decisions

Applied per-function during implementation and pinned by the plan:
- **`parseRangeSelection` moves with its clock parameterized** — its body does `toMs = Date.now()` (verified). It gains a `nowMs` argument (adapter supplies `Date.now()`), exactly as 2.2 parameterized `irrDecision`'s `nowMs`. Behavior-preserving; a golden vector pins a relative-range case with a fixed `nowMs`.
- **The SEVEN helper-bound dispatchers STAY in the adapter (HARD-DECISION #1):** `statusForCardValue`, `normalizeGatewayEui`, `uniqueDeveuis`, `displaySafeSourceKey`, `soilRowsHaveWarning`, `dendroRowsHaveWarning`, `environmentRowsHaveWarning` — each calls `osiHistory.*` (verified: `classifySoilStatus`/`classifyDendroStatus`/`classifyEnvironmentStatus`/`normalizeDeveui`). Moving any of them would either re-`require` the helper inside the glue module (avoidable coupling) or force the caller to inject the helper's functions (cosmetic gain). The pure glue module has NO `osi-history-helper` dependency (§E); the router keeps its own `osiHistory`-bound dispatchers adapter-local. **HARD-DECISION #1** (contested: alternative is to inject the classify/normalize fns; rejected for coupling). If a later item wants them under the contract, they move then — not here.
- **`parseRangeSelection` moves with `nowMs` parameterized** (verified `toMs = Date.now()`) — adapter supplies `Date.now()`, mirroring 2.2's `irrDecision`. It is the ONE genuine wall-clock reader in the MOVE set.
- **The wall-clock / adapter-state helpers STAY:** `nowIso` (`new Date().toISOString()` — it is the clock itself, adapter-facing), and `markPhase`/`addPhase` (verified: both call `Date.now()` AND mutate the adapter-scope `const historyPhases` object — closure over adapter state). Only `phaseSummary(phases)` (pure, arg-driven) moves.
- **A function moves only if it is context-free AND not a duplicate of an `osi-history-helper` export** (no `node.*`, no DB handle, no `osiDb`, no `osiHistory`, no `msg.`, no `Date.now()`/wall-clock unless parameterized, no outer-scope mutation). The §A "moves" list is the verified set (full-body scanned); the plan re-confirms each with a grep of the extracted body AND a check against the helper's `module.exports` before finalizing the move. Any function found impure that is not on the STAY lists above is reported and left in the adapter.
- **Async DB/helper functions do NOT move** — `getOwnedZoneContext`, `build*CardData`, workspace/preference CRUD, etc. stay; they are I/O + delegation orchestration. Their pure sub-computations (series shaping, calendar rows, preference maps) move and the adapter functions call `HR.<name>(...)`.

### C. The adapter size reality — honest against DD4's re-scoped bar

Per 2.2 §C (recorded DD4 refinement for I/O-heavy seams, inherited by 2.4), the residual `history-api-router-fn` — the largest node — cannot be 2 KB: the auth block, the `ensureHistoryTables` DDL, the many SQL reads, the `osiHistory.*` delegation, the workspace/preference CRUD, and the multi-route dispatch are irreducibly large. What the extraction commits to:
- **The pure router-glue core (the §A functions) leaves the node**, moving into `osi-history-router/index.js`. The residual `func` is auth + DDL + SQL I/O + `osiHistory.*` delegation + route dispatch + `osiLib.require('history-router')` + calls into the glue module. The **1.A2 total-JS scoreboard drops** by the moved char mass (module code lives in a helper file, not flows.json). Because this is the largest node, its extraction is the single biggest scoreboard win of the batch.
- **The re-scoped DD4 bar applies:** "the extracted pure glue is a standalone tested module; the residual node is auth + DDL + I/O + helper-delegation + HTTP orchestration only, with zero inline parsing/validation/shaping logic left." 4.2 does not re-open the DD4 refinement.
- The residual node still loads via `osiLib.require` (DD2/DD4), still gets its glue proven by golden vectors, still lowers the scoreboard. **A big-bang rewrite is explicitly forbidden (§Problem, program Risks):** the move is function-family-at-a-time, each pinned by a vector, the whole-node behavior pinned by the end-to-end replay.

### D. Golden vectors — captured from the CURRENT node before extraction (behavior-preservation LAW)

Golden vectors are captured by executing the **CURRENT** `history-api-router-fn` body against fixture DB rows + a fixed clock + the **real** (already-tested) `osi-history-helper`, for a set of representative routes, using the `rehearse-devices-rebuild.js` technique (§ground-truth 6). The captured input/output pairs are the contract the extracted module must reproduce exactly.

**Two vector layers, both captured before the extraction touches the node:**

1. **Unit vectors for the pure glue functions (the module's own `node --test`).** For each function in §A, capture representative `(inputs) → output` pairs — e.g. `parseZoneId(query)`, `parseRangeSelection(query, config, scopeContext, { nowMs })` (relative + absolute + season cases), `validateAggregation(value)`, `buildSeriesFromAggregate(aggregate, channelKey)`, `pointQuality(point)`, `buildPreferenceMap(rows)`, `normalizeWorkspaceRow(row)`, `seasonRangeForContext(...)`, `supportedRangesForCard(cardType)`, the source-classification predicates, `calendarRowsFromSeries(series)`. These become `osi-history-router/index.test.js` assertions — the module reproduces each output byte-for-byte.
2. **End-to-end golden vectors (the extraction proof).** A capture harness (`scripts/capture-history-router-vectors.js`, kept in `scripts/`, not shipped to the Pi) seeds a `node:sqlite` DB from `database/seed-blank.sql` plus fixture rows (a zone with soil/dendro/environment devices + history rows, a gateway, a workspace, preferences), pins the clock, loads the **real** `osi-history-helper` as `osiHistory` (deterministic, under 1.A3's test — NOT stubbed), executes the **current** node `func` against it via `makeFacadeShim` for a representative set of routes (a card-summary GET, a series/aggregate GET, a workspace CRUD op, a CSV export), and snapshots each returned response into `docs/contracts/history-router/` fixtures. Then the **extracted** node is run against the identical seed + clock + helper for the same routes and its outputs are asserted deep-equal to the snapshot. This proves the whole router, post-extraction, produces identical responses across routes — the strongest behavior-preservation evidence without a live gateway. (The CSV export route asserts the streamed CSV bytes are identical — pinning that the `osiHistory.buildZoneExportCsv`/`respondCsv` path was untouched.)

**Fixture location:** per-function unit vectors live beside the module (`osi-history-router/index.test.js` / a `fixtures/` subdir); the end-to-end route→expected-response fixtures live in `docs/contracts/history-router/` (this spec CREATES that directory + its first fixtures; no cross-repo mirror is defined by 4.2).

### E. Load mechanism (cite osi-lib spec §A/§B + §C HTTP rule + §E, do not re-derive)

The module loads via `osiLib.require('history-router')` per `2026-07-07-osi-lib-loader-design.md` §A/§B (the `osiLib.require` mechanism + `NAME_TO_PATH`) under §E's rule that every new seam MUST load this way: registered in `osi-lib`'s `NAME_TO_PATH`, packaged through the three delivery surfaces (runtime `package.json` + `package-lock.json`, `98_osi_node_red_seed` module loop, `deploy.sh` fetch pair), gated by `verify-helper-registration.js`. **The adapter's load-failure path follows osi-lib §C's HTTP-shaped rule (verified line 88):** on `!r.ok`, `msg.statusCode = 503; msg.payload = { error: 'module_unavailable', module: 'history-router', message: r.error }; return [null, msg]` — never a dead node on the request path. `osiLib` is added to the node's `libs` alongside the existing `osiDb`/`osiHistory`/`crypto`. `osi-history-router` becomes a new non-codec `NAME_TO_PATH` entry, so `verify-helper-registration.js` enforces its three-surface registration at merge time.

- **Moving the `osiHistory.*` delegation or the CSV export path** — stays adapter-side; `verify-sync-flow.js:1231` (`osiHistory.buildZoneExportCsv` must remain) stays satisfied.
- **The auth block** (`verifyBearer`/JWT) — stays adapter-side.
- **Moving any of the seven helper-bound dispatchers** (`statusForCardValue`, `normalizeGatewayEui`, `uniqueDeveuis`, `displaySafeSourceKey`, `soilRowsHaveWarning`, `dendroRowsHaveWarning`, `environmentRowsHaveWarning`) (§B, HARD-DECISION #1) — all stay adapter-side (helper-bound); not this item's concern.
- **Moving the wall-clock/adapter-state helpers** (`nowIso`, `markPhase`, `addPhase`) — stay adapter-side.
- **Any cross-repo contract gate** — 4.2 creates `docs/contracts/history-router/` fixtures for behavior-preservation but defines no osi-server mirror.
- **Deploying to any gateway.** No SSH, no live system.
- **Hitting a literal 2 KB adapter** (§C) — structurally impossible for the largest HTTP-request-path node; re-scoped per 2.2 to "zero inline glue logic remains."
