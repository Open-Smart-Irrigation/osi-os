# Sync-Ingest Hardening — Per-Event TX Boundary + Dead-Letter + Batch Cap + Rate Limit

**Status:** Draft
**Refactor-program item:** 1.B4 (DD13 + DD15)
**Focus: osi-server**
**Gate:** this item is the **hard gate for the Uganda gateway catch-up (#87)** — a weeks-stale gateway replaying backlog into today's single batch-wide `@Transactional` is a live production-outage scenario, not a hypothetical.
**Depends on:** 1.B3 (CI + `./gradlew test` wiring) for automated enforcement; this spec's own tests must still pass locally (`./gradlew test`) before 1.B3 lands.

## Problem

`EdgeSyncService.applyEventsV2` (`backend/src/main/java/org/osi/server/sync/EdgeSyncService.java:212-235`) wraps an entire incoming batch in one `@Transactional` method and loops calling the private `applyEventV2` for each event. If any JPA/repository call inside that loop throws past the per-event `try/catch` (`EdgeSyncService.java:292-318` only catches around `applyEvent`+watermark+inbox-save; a failure elsewhere — e.g. in `ownershipService.requireMutate`, or a constraint violation Hibernate defers to flush time — propagates out of `applyEventV2` and up through `applyEventsV2`), Spring marks the transaction rollback-only. The whole batch then fails at commit: every `SyncInboxEvent` dedup row and domain mutation from events `1..k-1` that "succeeded" earlier in the same loop is rolled back too. The edge does not know this happened differently from a clean failure — it just re-sends the same 100-event batch next cycle, hits the same poison event at the same position, and repeats forever. A weeks-stale gateway (Uganda, #87) replaying backlog is exactly the scenario that produces one.

Compounding this: `SyncInboxEvent` (`SyncInboxEvent.java`) has only `event_uuid`, `processed_at`, `source_node` — no status, reason, or payload. A rejected event's row is byte-identical to an applied one, and the payload that caused the rejection is gone. There is no forensic trail for "what did gateway X send that we dropped, and why," and no batch-size/rate ceiling on `/api/v1/sync/edge/events` to bound how much work one gateway's backlog can push onto the shared VPS.

## Verified ground truth (corrections to the expert report)

The cloud-backend expert report (`analysis/refactor-program-2026/expert-cloud-backend.md`) is accurate on the transaction hazard and the inbox gap. Two claims needed direct verification and are corrected here:

1. **No `SUPPORTED_OPS` gate exists.** The report says unsupported ops are caught by "a `SUPPORTED_OPS` gate returning `rejected(…, "unknown_op")`." There is no such named constant or set anywhere in `EdgeSyncService.java`. The actual mechanism is a dry-run call to the same op-dispatch method: `applyEventV2` calls `applyEvent(gatewayDeviceEui, event, true)` (dry-run) before ownership/watermark checks (`EdgeSyncService.java:254`); the op-switch's `default` branch (`EdgeSyncService.java:480-485`) returns `false` in dry-run mode instead of throwing, and `applyEventV2` translates that into `rejected(event.eventUuid(), "unknown_op")`. Functionally equivalent to a gate, but there is no enumerable `SUPPORTED_OPS` set to extend — the switch's case list *is* the gate. This spec's dead-letter write path must hook the same call site, not a nonexistent constant.
2. **No cross-repo schema mirror or parity check exists today.** `docs/contracts/sync-schema/README.md` states mirrored copies in osi-server "must match bytewise," and `scripts/verify-sync-contract.js` runs only in osi-os — it checks the flows.json command registry against `commands.schema.json` and that the three schema files parse as valid JSON. It does **not** compare against any copy in osi-server, because **no such copy exists** (`find osi-server -iname '*.schema.json'` returns nothing outside `frontend/node_modules`). The "cross-repo parity gate" that DD13's runtime-validation clause presumes is aspirational (Tranche B / DD14 territory per `2026-06-30-sync-contract-package-design.md` §7), not live infrastructure. This directly drives Decision F below (defer schema validation).

Everything else load-bearing checks out: `applyEventV2`'s reject-reason taxonomy (`missing_event_uuid`, `unknown_op`, `ownership_denied:*`, `stale_sync_version`, `equal_version_payload_conflict`, plus the `IllegalArgumentException` message from `applyEvent`, plus `retryable` for any other `Exception`), the watermark idempotency table, the `contract_version` mismatch-logged-not-rejected behavior, and the retention-job precedent (`SyncInboxRetentionJob`, `@Scheduled` + `@Value`-defaulted days, no `application.yml` entry required) are all confirmed directly in code.

**Edge-side facts confirmed** (`conf/full_raspberrypi_bcm27xx_bcm2712/files/usr/share/flows.json`), load-bearing for the response contract:
- `Build Edge Event Batch` / `Run Force Sync`: `SELECT ... FROM sync_outbox WHERE delivered_at IS NULL AND rejected_at IS NULL ORDER BY occurred_at ASC LIMIT 100` — hard-coded batch cap of **100**, not configurable.
- `Flush Sync Outbox` inject node: `repeat: "30"` (seconds), `onceDelay: "15"` — steady-state cadence is **one batch of ≤100 events every 30 s** per gateway.
- `Mark Synced Events Delivered`: `APPLIED`/`DUPLICATE` → `delivered_at` set; `REJECTED` → `rejected_at` + `rejection_reason` set (the edge already persists the reject reason locally); `RETRYABLE_ERROR` → **no row write at all**, so the row is naturally re-selected next cycle. No attempt counter is read or incremented anywhere (`retry_count` column exists in DDL, is never touched by flows.json).
- Non-2xx HTTP response or thrown network error on the whole POST: the same guard (`msg.statusCode < 200 || msg.statusCode >= 300`, and a caught network error is normalized to `statusCode = 0` which fails the same check) leaves **every row in the batch untouched** — nothing is dead-lettered edge-side, the entire batch is simply retried next tick. **This means a 429 from a new rate limit is automatically safe** — it is indistinguishable, from the edge's perspective, from any other transient failure it already tolerates.
- `sync_outbox.last_retryable_failure_at TEXT` **does exist** in the schema (`database/migrations/ordered/0001__baseline.sql:471`, `seed-blank.sql:538`) — correcting an intermediate research pass that reported the column absent. It is present in DDL but flows.json never reads or writes it; it is reserved/dead today, not a live backoff signal. Any change to make the edge use it is out of scope (edge-side changes are a non-goal here).

## Design

### A. Per-event transaction boundary

**Mechanism: extract a package-private `SyncEventTxExecutor` bean** carrying three narrow transactional methods — `applyOne(...)` (`@Transactional(propagation = Propagation.REQUIRES_NEW)`), `recordRejection(...)` (`REQUIRES_NEW`, see below), `finalizeBatch(...)` (`@Transactional`) — called by `EdgeSyncService.applyEventsV2`'s loop instead of the current same-bean private-method call.

Rejected alternatives:
- *Self-injection* (`@Autowired private EdgeSyncService self`, call `self.applyEventV2(...)`) — works (goes through the proxy) but is a well-known code smell that surprises the next reader of a file already flagged for a future split (DD12); the executor extraction is a strict subset of that future split, so it's free groundwork rather than throwaway.
- *`TransactionTemplate` inlined in `applyEventsV2`* — no new bean, but keeps the transactional unit's boundaries implicit inside a loop body in a 1800-line file; a named bean with narrow methods is easier to unit-test in isolation (construct it directly with mocked repositories, no Spring context needed) and easier to find when DD12's per-resource appliers eventually replace its body.

**Why this doesn't fight DD12:** `SyncEventTxExecutor.applyOne(...)` takes exactly the arguments `applyEventV2` takes today and returns exactly `SyncEventResult`. When DD12 later splits the op-switch into per-resource appliers, the executor's job stays "one event, one new transaction, one result" — the appliers slot in *inside* `applyOne`, unchanged at the boundary. This spec does not attempt that split; `applyOne`'s body is today's `applyEventV2` body, moved, not rewritten.

**What commits atomically per event, inside `applyOne`:**
- The domain mutation (the op-switch's `upsert*`/`insert*` call) — unchanged from today.
- The `SyncResourceWatermark` upsert — unchanged from today.
- The `SyncInboxEvent` dedup row (`saveInboxTerminal`) — unchanged from today for in-method terminal paths (applied and the validation-layer rejects below).
- **New:** the `sync_dead_letter` row for in-method rejects, inside the same transaction as the inbox row — one commit, both rows exist or neither does.
- **New:** an explicit `entityManager.flush()` as the last statement of `applyOne`'s happy path, so Hibernate's deferred constraint checks surface inside the per-event scope instead of at the proxy-boundary commit.

**Where exceptions surface — the load-bearing subtlety.** Hibernate defers flush to commit, and under `REQUIRES_NEW` the commit happens at the *proxy boundary as `applyOne` returns* — after any in-method `try/catch` has gone out of scope. So the failure class the Problem statement names (a constraint violation deferred to flush time) cannot be fully classified inside `applyOne`. The design distinguishes two failure surfaces:

1. **In-method, transaction still healthy:** failures thrown by `EdgeSyncService`'s *own* frames — missing UUID, unknown op (dry-run returns `false`, no exception), `OwnershipDeniedException` (thrown by `requireMutate`'s own code, not from inside a repository proxy), stale/equal watermark comparisons, and `IllegalArgumentException` from payload validation (e.g. "Zone not found", thrown by our `orElseThrow` after the repository returned). These do not mark the transaction rollback-only; `applyOne` classifies them in-method and writes the inbox + dead-letter rows in the same, still-committable transaction — today's semantics plus the dead-letter row. **`applyOne` removes today's blanket `catch (Exception) → retryable`:** an exception originating *inside* a repository/JPA call (constraint violation at the explicit flush, deadlock, timeout) has already marked the participating transaction rollback-only, and swallowing it would only convert the failure into an `UnexpectedRollbackException` at the proxy commit. Such exceptions propagate out of `applyOne`.
2. **At the loop, transaction already rolled back:** `applyEventsV2`'s loop wraps every `txExecutor.applyOne(...)` call in `try/catch` — **the loop is where flush-time and commit-time exceptions surface**, and this catch is load-bearing: without it, event k aborts the batch with an HTTP 500, `k+1..n` never process, and the edge re-sends the identical batch forever (events `1..k-1` would survive their committed `REQUIRES_NEW` transactions, but batch progress would still wedge at k). The catch runs the exception classifier (§B). Permanent classes → `txExecutor.recordRejection(gatewayEui, sourceNode, event, reason, detail)`: a second, tiny `@Transactional(propagation = REQUIRES_NEW)` method on the executor that writes the dead-letter row *and* the inbox terminal row in a fresh, healthy transaction — the only way a flush/commit-time-rejected event gets both rows, since its own transaction is gone. Transient classes → nothing persisted, result `retryable(...)`. `recordRejection` is invoked from the loop (through the bean proxy), never from within `applyOne` — an in-bean call would be the same self-invocation trap that motivated the executor; it also keeps the classifier at a single call site.

**Batch orchestration (`applyEventsV2`) becomes:**
```
// applyEventsV2 is NOT @Transactional — see below
for event in request.events():
    try:
        result = txExecutor.applyOne(gatewayEui, sourceNode, event)  // REQUIRES_NEW; commits or rolls back alone
    catch (ex):
        if classify(ex) is PERMANENT:                                // §B classifier
            txExecutor.recordRejection(gatewayEui, sourceNode, event, reason(ex), detail(ex))  // fresh REQUIRES_NEW
            result = rejected(event.uuid, reason(ex))
        else:
            result = retryable(event.uuid, detail(ex))               // nothing persisted
    results.add(result)
txExecutor.finalizeBatch(sourceNode, gatewayEui, lastEventAt, lastEventUuid)  // one small @Transactional
```

**Two hardening rules on the rejection path:** if `recordRejection` itself throws (e.g. the DB is momentarily unavailable), the loop downgrades that event to `RETRYABLE_ERROR` instead of failing the batch — never silently drop; and for an `event_uuid` longer than 36 characters (unstorable in `sync_inbox.event_uuid VARCHAR(36)`), `recordRejection` skips the inbox row — dedup for that event rides on `sync_dead_letter`'s unique index instead.

**`applyEventsV2` loses its `@Transactional` entirely** — it does not keep a thin outer one. Two reasons: (1) **connection double-checkout** — an outer transaction holds one Hikari connection for the whole batch while every `REQUIRES_NEW` call suspends it and checks out a *second*; the repo has **no Hikari pool configuration anywhere** (verified — Spring Boot default `maximumPoolSize = 10`), so ~5 concurrent batch POSTs (30 gateways on aligned 30 s ticks; there is no jitter) could starve or deadlock the pool — the classic `REQUIRES_NEW` hazard. (2) An outer transaction would hold one idle-in-transaction session open across up to 100 inner commits — precisely the long-transaction behavior this item exists to eliminate. The `SyncCursor` save and `touchGatewayLastSeen` move into `txExecutor.finalizeBatch(...)`, one small `@Transactional` method called once after the loop — safe because the cursor is overwritten wholesale on the next successful batch and is never read for correctness during apply. With no outer transaction, `applyOne`'s `REQUIRES_NEW` has nothing to suspend (no double checkout in the shipped shape); the annotation stays `REQUIRES_NEW` rather than plain `@Transactional` so the per-event boundary survives any future transactional caller.

**Behavior when event k fails:** events `1..k-1` already committed (each in its own transaction) stay committed regardless of what happens to k. Event k is classified — in-method where its transaction is still healthy, at the loop where it is not — and either its terminal rows commit (in-tx for validation rejects; via `recordRejection` for flush/commit-time permanent faults) or nothing persists (retryable). Events `k+1..n` proceed in their own fresh transactions, unaffected. This is the fix: today k's failure poisons `1..n`; after this change k's failure affects only k, on every failure surface including commit time.

### B. Result-class semantics under the new boundary

| Result | Persisted | Response to edge | Edge behavior (verified) |
|---|---|---|---|
| **applied** | Domain mutation + watermark + inbox row. No dead-letter row. | `APPLIED` | `delivered_at` set; never re-sent. |
| **duplicate** | Inbox row only (already existed, or watermark-equal-hash case) — unchanged from today. No dead-letter row. | `DUPLICATE` | `delivered_at` set; never re-sent. Correct — a duplicate is not a failure. |
| **rejected — in-method** (`missing_event_uuid`, `unknown_op`, `ownership_denied:*`, `stale_sync_version`, `equal_version_payload_conflict`, or an `IllegalArgumentException` from validation) | Inbox row (dedup) **+ new `sync_dead_letter` row**, same transaction. Two per-reason deltas: `ownership_denied` today returns rejected *without* an inbox row — this design adds one, deliberately aligning it with every other terminal reject (safe: the edge already treats `REJECTED` as terminal); `missing_event_uuid` gets **no** dead-letter row (`event_uuid` is NOT NULL by design; the case is unreachable from a well-formed edge — `sync_outbox.event_uuid` is its PK). | `REJECTED` (unchanged wire shape) | `rejected_at` + `rejection_reason` set; edge never re-sends it. This is exactly today's edge behavior — it already treats `REJECTED` as terminal. |
| **rejected — flush/commit-time, permanent** (new class: `integrity_violation`) | Inbox row + dead-letter row, written via `recordRejection` in a **fresh** `REQUIRES_NEW` transaction (the event's own transaction already rolled back). | `REJECTED` | Same as above — terminal at the edge. |
| **retryable** (transient fault classes, and the unknown default) | **Nothing** — no inbox row, no dead-letter row, matching today. | `RETRYABLE_ERROR` (unchanged) | No row write; naturally re-selected by next `LIMIT 100` poll. Re-sendable without tripping dedup, because no inbox row exists for it — unchanged from today. |

**The exception classifier** (used by the loop's catch in §A; one small pure function, unit-testable without Spring):
- `org.springframework.dao.DataIntegrityViolationException` or a Hibernate `ConstraintViolationException` in the cause chain → **REJECTED**, new reason **`integrity_violation`** — permanent: the event's data violates the mirror schema and will violate it identically on every retry.
- `TransientDataAccessException`, `CannotAcquireLockException`, `QueryTimeoutException`, `DeadlockLoserDataAccessException` → **RETRYABLE**, nothing persisted — these genuinely resolve on retry.
- Anything else (including `UnexpectedRollbackException` from a swallowed inner failure, and any exception the classifier does not recognize) → **RETRYABLE** — the conservative default: never permanently drop an event on uncertainty. A misclassified-permanent fault then loops visibly (edge re-sends every 30 s, server logs the same warning) rather than silently losing data, and the fix is to teach the classifier the new permanent class.

**Decision: rejected → immediate dead-letter, no attempt-counter/transition state.** With the classifier in place, the named poison class — a constraint-violating event from a stale gateway — lands in `rejected/integrity_violation` and is dead-lettered *immediately*, on first contact. Unbounded, backoff-free retry (today's edge behavior, unchanged) is therefore genuinely reserved for transient faults that retry actually fixes — deadlocks, lock timeouts, momentary DB pressure — which is exactly what unbounded retry is the correct policy for. This spec does not add an attempt counter to `SyncInboxEvent` or a retryable→dead-letter escalation path: (1) the inbox has no attempt counter today and adding one widens the hot dedup-check row the cloud expert explicitly warned against (§D3 of the expert report); (2) with permanent faults already classified out on first contact, an escalation path could only fire on a *misclassified or persistent-transient* fault — and auto-dead-lettering after N attempts would permanently drop an event that a server-side fix (or the transient clearing) would have applied. If a retryable event is observed looping in production, the remedy is to extend the classifier's permanent list — one line — not to add attempt-counting machinery (YAGNI).

**One inherent watermark consequence (documented in the §G tests):** a RETRYABLE event whose *successors* (same resource, higher `sync_version`) apply in a later batch will be `stale_sync_version`-rejected — and therefore dead-lettered — when its verbatim re-send finally arrives: the per-resource watermark has moved past it. Benign for last-writer-wins upserts (the newer state already won); append-only telemetry uses per-row watermark slots and never collides.

**`saveInboxTerminal` is kept as-is; the dead-letter row is added alongside it, not merged into it.** This directly follows the cloud expert's recommendation (§D3: "as a separate `sync_dead_letter` table, not widening the hot inbox row that every dedup check hits"). `SyncInboxEvent` stays a 3-column existence check; `sync_dead_letter` carries the forensic payload. For in-method rejects, `applyOne` writes both rows in its own transaction; for flush/commit-time permanent faults, `recordRejection` writes both rows in its fresh transaction — in every case the two rows commit together or not at all.

### C. `sync_dead_letter` table

New Flyway migration `V2026_07_07_001__sync_dead_letter.sql` (highest existing is `V2026_07_06_001__agroscope_shadow.sql`; next date-versioned slot, additive only, no destructive statements — creates one new table, no ALTER on any existing table).

```sql
CREATE TABLE sync_dead_letter (
    id                BIGSERIAL PRIMARY KEY,
    event_uuid        VARCHAR(64) NOT NULL,
    gateway_eui       VARCHAR(32) NOT NULL,
    source_node       VARCHAR(100),
    op                VARCHAR(50),
    aggregate_type    VARCHAR(50),
    aggregate_key     VARCHAR(100),
    payload           JSONB,
    reason            VARCHAR(50) NOT NULL,
    reason_detail     TEXT,
    contract_version  BIGINT,
    occurred_at       TIMESTAMPTZ,
    received_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX ux_sync_dead_letter_event_uuid ON sync_dead_letter (event_uuid);
CREATE INDEX ix_sync_dead_letter_gateway_received ON sync_dead_letter (gateway_eui, received_at DESC);
CREATE INDEX ix_sync_dead_letter_received_at ON sync_dead_letter (received_at);
```

- **`event_uuid` width 64, not the inbox's 36** — a malformed >36-char uuid is itself dead-letter-worthy (it is unstorable in `sync_inbox.event_uuid VARCHAR(36)` and is exactly the flush-time poison the §G repro uses); precedent: `sync_resource_watermarks.last_event_uuid` is already `VARCHAR(64)`. The third index `ix_sync_dead_letter_received_at` serves the retention job's pure `received_at < ?` scan, which the composite `(gateway_eui, received_at)` cannot.

- **`reason`** — varchar, not a Postgres enum, per house convention (`DeviceType` is "a string-constant class, not an enum" — `AGENTS.md` Conventions; `command_status` is the one place a native Postgres enum type is used, and that predates this convention). Values are the existing reject-reason strings verbatim plus the new flush/commit-time class (`missing_event_uuid`, `unknown_op`, `ownership_denied`, `stale_sync_version`, `equal_version_payload_conflict`, **`integrity_violation`** from the §B classifier, `invalid_payload` for `IllegalArgumentException` messages that don't fit a fixed string — store the exception message in `reason_detail`, keep `reason` a short stable code). This keeps `reason` groupable/queryable while `reason_detail` keeps the free-text detail (e.g. the specific ownership-denial message or `IllegalArgumentException#getMessage()`).
- **`payload`** — `jsonb` via `@JdbcTypeCode(SqlTypes.JSON)` + `columnDefinition = "jsonb"` + `Map<String, Object>`, the exact pattern already used by `DeviceCommand.payloadJson` and `SyncOutboxEvent.payloadJson`. Stores `event.payload()` (the raw map, including `contract_version`), not the `payloadWithOp`-augmented version — the raw wire payload is the forensic artifact; op/aggregate fields are already broken out into their own columns.
- **Uniqueness vs re-sent duplicates of an already-dead-lettered event:** `event_uuid` is unique. Because the edge sets `rejected_at`/`rejection_reason` locally on a `REJECTED` response and never re-sends that row (verified above), a genuine re-send of the same `event_uuid` after dead-lettering should not happen in normal operation — but the DB constraint is the actual guarantee, not the edge's good behavior. Both write paths (`applyOne` in-tx and `recordRejection`) upsert (`INSERT ... ON CONFLICT (event_uuid) DO NOTHING` via a repository method, or a pre-check via `existsById` mirroring the existing `inboxRepository.existsById` check) so a hypothetical re-send is a silent no-op, not a constraint-violation exception that would itself need retryable-classification.
- **Retention: 90 days**, a `SyncDeadLetterRetentionJob` following the exact `SyncInboxRetentionJob` shape (`@Scheduled` cron + `@Value("${osi.retention.sync-dead-letter.days:90}")`, no `application.yml` edit required — matching the existing pattern where `osi.retention.*` keys are Java-side defaults only). 90 days (not the inbox's 365) because the dead-letter table exists specifically to answer "what did we drop and why" during active triage of a slowly-upgrading fleet; the inbox's 365-day figure is justified by *replay-safety* (a V1 edge can still replay events a year old and needs the dedup row to exist), which does not apply to dead-letters — there is nothing to replay-protect once an operator has seen and acted on (or explicitly ignored) a rejection. Start generous per the expert report's explicit warning that dead-letter retention is a one-way door on forensic value; tighten only after the admin surface (§D) has been used in anger.
- **Relationship to `saveInboxTerminal`:** unchanged, see §B — kept for dedup, dead-letter added alongside, never merged.

### D. Admin/observability surface

**List endpoint, read-only, no requeue action** (requeue is YAGNI: there is no reprocessing pipeline to feed it, and every current `rejected` reason is a permanent classification — requeueing a `stale_sync_version` or `unknown_op` event would fail identically; building requeue now would be speculative machinery for a use case that doesn't exist yet).

Follows the exact `SyncHealthController` precedent (`/api/v1/admin/sync-health`, `@PreAuthorize("hasAnyRole('ADMIN','SUPER_ADMIN')")`, bounded `limit` query param):

```
GET /api/v1/admin/sync-dead-letters?gatewayEui=<eui>&limit=100
```
- `gatewayEui` optional (omit for fleet-wide, matching `SyncHealthController`'s pattern); `limit` bounded `[1, 500]` identically.
- Returns dead-letter rows ordered `received_at DESC`, mapped through a `SyncDeadLetterResponse` DTO (not the entity directly — avoids ever leaking Postgres columns not meant for the wire, and matches the existing `PendingCommandResponse`/`FailedCommandResponse` DTO convention already used by `EdgeSyncController`).

**Counters:** `sync_dead_letter_total` (fleet-wide `COUNT(*)`, or per-gateway when `gatewayEui` is supplied) and per-batch `batch_events_applied` / `batch_events_rejected` counts. Exposure path — keep minimal, per the instruction:
- `sync_dead_letter_total` (following the allowlist's `<table>_total` naming convention) slots into the existing `DbHealthCounters.ALLOWED_COUNT_SOURCES` allowlist (`org.osi.server.retention.DbHealthCounters`) exactly like `sync_inbox`/`sync_outbox` today — one new allowlisted source string (`"sync_dead_letter"`), one new `count(...)` call. This is already exposed via whatever consumes `DbHealthCounters.snapshot()` today (verify at implementation time; out of scope to change that consumer here).
- `batch_events_applied`/`batch_events_rejected` as **Micrometer counters** (`Counter.builder("osi.sync.batch.events").tag("result", "applied"|"rejected"|...)...`) incremented once per `SyncEventResult.Status` inside `applyOne` — but **only if Micrometer's registry is already on the classpath by the time this ships**. Today it is not (`build.gradle.kts` has `spring-boot-starter-actuator` but no `micrometer-registry-*`; confirmed absent). Adding a full metrics registry is item 1.B3's job ("Micrometer/actuator metrics endpoint"), which this item depends on. **Decision: if 1.B3 has landed Micrometer by the time this ships, wire the counters there (near-zero marginal cost); if not, ship this item without them and note it as a one-line follow-up** — do not block the transaction/dead-letter/rate-limit work (the actual outage-prevention content of this spec) on a metrics dependency that belongs to a different roadmap item. The `dead_letter_total` DB counter above is independent of Micrometer and ships regardless.

### E. Batch-size cap + gateway-scoped rate limit on `/api/v1/sync/edge/events`

**Batch-size cap: reject (400) any protocol-2 request with `events.size() > 100`; the legacy V1 path is left uncapped.** V1 edge batch sizes are unverifiable from the repo, a 400-wedge on a legacy gateway would be worse than an unbounded-but-still-rate-limited legacy batch, and the #87 catch-up runs on current protocol-2 flows after a full deploy. The edge already caps itself at 100 per `LIMIT 100` in `Build Edge Event Batch` — this is a server-side backstop against a misbehaving or future edge build, not a change to normal traffic. 100 is chosen to exactly match the edge's own cap rather than invent a second number that could someday diverge from it. Enforced in `EdgeSyncController.applyEvents` before calling `applyEventsV2` (fail fast, no DB work at all) — return `400 Bad Request` with a JSON body naming the limit. One honest caveat on the 400-vs-429 distinction: the edge treats **every** non-2xx identically (leave rows untouched, retry next tick — verified above), so a hypothetical >100 batch would retry forever against the 400 exactly as it would against a 429; the distinct status aids server logs and operator triage, not edge behavior. Such a wedge is edge-visible through `sync_outbox` pending-count growth and heartbeat, and needs operator intervention — it is not self-healing. Since the edge never produces a >100 batch today, this path is defense-in-depth, not a live constraint on Uganda's catch-up.

**Rate limit: 10 requests/minute per gateway EUI on `/api/v1/sync/edge/events`**, reusing the `RateLimitFilter` Bucket4j pattern (`org.osi.server.security.RateLimitFilter` — direct `Bucket4j` instantiation, bounded in-memory `LinkedHashMap` keyed by identity with an LRU eviction cap, no Spring bean). **Keyed by gateway EUI** (extracted from the already-validated JWT sync-token claim, the same `tokenGatewayDeviceEui` the controller already resolves in `authorizeGateway` — not by IP, since the requirement is per-gateway fairness and gateways may share NAT/proxy IPs in the field).

**Arithmetic for headroom:** steady-state cadence is one ≤100-event batch per 30 s = 2 requests/minute/gateway. A backlog-draining gateway (Uganda-shaped, weeks stale) will *want* to burst — its `Flush Sync Outbox` inject node still fires every 30 s regardless of backlog size, so it already self-throttles to at most 2 req/min even mid-catch-up (each request still capped at 100 events by its own query). **10 req/min is 5× the steady-state rate and does not bind under the edge's automatic behavior** — it exists as a backstop against a hypothetical future edge change (e.g. a tighter poll interval), a misbehaving/compromised gateway, or a runaway operator force-sync loop (quantified below), not as a throttle Uganda's automatic drain will ever hit. Backlog-drain throughput is therefore governed by the 30 s cadence × 100 events, not by this limit: a gateway with, say, 3 weeks offline at a typical low-frequency sensor cadence (dendro/chameleon/device_data appends across a handful of zones) produces on the order of a few thousand to low tens of thousands of backlog events — at 100 events/30 s = 200 events/min = 12,000/hour, that drains in under a few hours, comfortably "hours not days," without the rate limit ever engaging. If a future gateway's backlog volume turns out to be an order of magnitude larger than this estimate, that is a capacity question for the edge's own flush cadence, not something this rate limit should be loosened to compensate for — the number is chosen to be a no-op under known behavior, not a throughput lever.

**Force Sync interaction (verified in `Run Force Sync`, node `sync-force-build`):** the force-sync node makes exactly **one** POST to `/api/v1/sync/edge/events` per invocation, using the *same* drain query verbatim (`WHERE delivered_at IS NULL AND rejected_at IS NULL ORDER BY occurred_at ASC LIMIT 100`) — its other HTTP calls hit `/edge/bootstrap` and `/pending-commands`, which this rate limit does not touch. An operator repeatedly triggering force-sync during the #87 catch-up window stacks with the 30 s tick, and that stacking is the one realistic way to approach the limit: at the full 10 req/min ceiling the drain rate is 10 × 100 = **1,000 events/min ≈ 60,000/hour** — comfortably acceptable catch-up pacing (a multi-week backlog clears within the hour at that ceiling), so the limit does not meaningfully slow an operator-assisted drain either. Force-sync's non-2xx handling is verified equivalent to the tick path's: on any non-2xx the node writes **nothing** to `sync_outbox` (`afterCount = beforeCount`) — same leave-untouched, retry-safe semantics — but unlike the silent tick, it *surfaces* the failure to the operator (`summary.outbox.error` in the force-sync result and `recordFailure('outbox', …)` into `sync_state.lastError`). A 429 during a rapid force-sync burst mid-catch-up therefore shows up as a failed outbox phase in the GUI: cosmetic, data-safe noise the #87 runbook should expect and note, not a defect.

**429 response and edge interaction:** a rejected request due to rate limit returns `429 Too Many Requests`, the same status `RateLimitFilter` already returns elsewhere in this codebase, with the same plain JSON error body shape. Confirmed safe: the edge's `Mark Synced Events Delivered` guard treats **any** `statusCode < 200 || statusCode >= 300` — including 429 and the same-shape 400 above — as "touch nothing, retry the same batch next 30 s cycle." A 429 is therefore indistinguishable, from the edge's perspective, from any other transient failure it already tolerates today; no edge-side change is needed and none is made.

### F. Runtime JSON-Schema validation at ingest

**Decision: defer to the contract-fixtures item (Phase 3, program item "Contract fixtures... validated in both repos' CI"), explicitly out of scope here.** DD13's charter line includes it, but §"Verified ground truth" above establishes that **no schema copy or parity mechanism exists in osi-server today** — `events.schema.json` lives only in `docs/contracts/sync-schema/` in osi-os, and `verify-sync-contract.js` never compares it against anything in osi-server because there is nothing there to compare against. Shipping runtime validation in this item would mean either (a) hand-copying `events.schema.json` into osi-server with no CI check that the copy stays byte-identical — reintroducing exactly the drift class DD13 exists to close, just one layer up — or (b) building the packaging/parity mechanism itself, which is squarely the cross-repo contract package's job (`2026-06-30-sync-contract-package-design.md` Tranche B, and the refactor program's Phase 3 "Contract fixtures... validated in both repos' CI", depending on 1.B3's CI). Building a one-off, unverified copy now would be worse than not validating at all: a silently-stale schema gives false confidence. **When the parity mechanism lands, its dead-letter hook is simple and is recorded here for that item to pick up:** validate the incoming event's `payload` against the packaged `events.schema.json` before the op-dispatch switch in `applyOne`; on failure, write a `sync_dead_letter` row with `reason = 'schema_violation'` and `reason_detail` = the validator's error path/message, and return `rejected(event.eventUuid(), "schema_violation")` — never a batch failure, consistent with every other rejection path in this design. No code for this ships in 1.B4.

### G. Testcontainers poison-batch reproduction (DD15)

Scoped exactly per DD15: **Testcontainers Postgres 16, single reused container**, for the sync/Flyway path only; everything else in the codebase stays Mockito. New test source root or package: `org.osi.server.sync` test classes annotated `@Testcontainers` with a `static final PostgreSQLContainer<?>` shared across the class (JUnit 5 `@Container` static field — reused across test methods in one run, not started per-test), Flyway clean-migrate against it via the actual `db/migration` directory (no H2, per AGENTS.md: "h2 dep exists but no Postgres-backed tests" and the expert report's "H2 ≠ Postgres" rejection of Option A).

**Spring context strategy — a focused JPA slice, not `@SpringBootTest`.** Full-context boot would run `DataInitializer`, which **fails hard** without an enabled super admin / `SUPERADMIN_*` bootstrap credentials, and would drag in MQTT/WebSocket/security config the sync path doesn't need. Instead: `@DataJpaTest` with `@AutoConfigureTestDatabase(replace = Replace.NONE)` pointed at the container's datasource (Flyway enabled so the real migrations run), `@Import`ing `EdgeSyncService` + `SyncEventTxExecutor`, with non-JPA collaborators (`EdgeOwnershipService`, command/lease services, zone environment/soil/linked-account services, the `ApplicationEventPublisher` consumer side) provided as `@MockBean`s. Real JPA repositories + real transactions + real Postgres; mocked everything that isn't the surface under test.

**Repro test (proves today's bug, then proves the fix):**
1. Seed a batch of N events (N=5 is enough to demonstrate) where event `k` (e.g. k=3) is constructed to throw inside a repository call — e.g. a `DEVICE_DATA_APPENDED` event whose payload is shaped to violate a NOT NULL/FK constraint that only surfaces at flush time, past the existing `IllegalArgumentException` guard, or a mock/wrapped repository that throws `DataIntegrityViolationException` on the k-th call.
2. **Before the fix (documents the bug, run once during development to confirm, not a permanent assertion of broken behavior):** calling today's `applyEventsV2` on this batch rolls back the whole transaction; assert (via a raw JDBC count against the container) that `SyncInboxEvent` rows for events `1..k-1` do **not** exist after the call — proving the whole-batch loss.
3. **After the fix (the permanent CI assertion):** calling the new `applyEventsV2` (delegating per-event to `SyncEventTxExecutor`) on the same batch commits `1..k-1` (assert their inbox rows and domain rows exist), classifies `k` per §B — for the constraint-violation fault, assert a `sync_dead_letter` row with `reason = 'integrity_violation'` **and** an inbox terminal row (both written by `recordRejection`, proving the fresh-transaction path); add a second variant where `k` throws a transient class (e.g. a wrapped `CannotAcquireLockException`) and assert **no** rows for `k` plus a `RETRYABLE_ERROR` result — and applies `k+1..n` (assert their rows exist too). This is the permanent regression test, covering both classifier branches at the loop boundary.

**Synthetic backlog-drain test:** 5,000–10,000 events with a small injected poison mix (e.g. 1 in 500 malformed) run through the new `applyEventsV2` path in batches of 100 (mirroring the edge's real batching), asserting: (a) total applied + rejected + retryable + duplicate counts reconcile to the input count, (b) every poison event is dead-lettered or left retryable per its fault class, never silently dropped, (c) no non-poison event in the same or adjacent batches is affected by a poison event's presence (the cross-event-contamination check that is the entire point of this spec), (d) wall-clock time stays within a sane bound for a local Testcontainers run (a smoke assertion, not a formal perf benchmark — e.g. under 60 s for 10k events on CI hardware; tune once the real number is observed, don't guess a strict SLA here).

**Run path:** `./gradlew test` locally, today — this does not depend on CI existing. Docker must be available locally for Testcontainers to start the container; if the developer's environment lacks Docker, this test class fails at container-startup with a clear Testcontainers error rather than a cryptic one, which is an acceptable local-dev constraint already implied by DD15. Wiring these tests into a CI workflow (GitHub Actions with Docker available) is item 1.B3's job — noted as a dependency, not built here.

## Open decisions

None outstanding.

- Transaction mechanism: **new `SyncEventTxExecutor` bean**, decided in §A (over self-injection or an inlined `TransactionTemplate`), because it shapes cleanly into DD12's future split without extra rework.
- Flush/commit-time exception handling: **loop-level catch + exception classifier + `recordRejection` in a fresh `REQUIRES_NEW` tx**, decided in §A/§B — the `REQUIRES_NEW` commit fires at the proxy boundary, after `applyOne`'s in-method scope, so the loop is the only place commit-time faults can be classified; `recordRejection` is loop-invoked (through the proxy) because an in-bean call would repeat the self-invocation trap.
- Outer batch transaction: **removed entirely** (not thinned), decided in §A — `REQUIRES_NEW`-under-outer-tx double-checks-out Hikari connections (no pool config exists; default max 10; 30 gateways on unjittered 30 s ticks) and holds an idle-in-transaction session across up to 100 inner commits; cursor + last-seen bookkeeping moves to `finalizeBatch`.
- Retryable escalation to dead-letter: **not built**, decided in §B — with the classifier, permanent faults (including the flush-time poison class, `integrity_violation`) dead-letter on first contact, so unbounded retry is genuinely reserved for transient faults; the unknown-exception default is RETRYABLE (loop visibly rather than drop silently), and the remedy for a misclassification is a one-line classifier extension, not attempt-counting machinery on the hot inbox row.
- Schema validation scope: **deferred to the contract-fixtures item**, decided in §F, because the cross-repo parity mechanism DD13 presumes does not exist yet — verified directly, not assumed.
- Micrometer counters: **conditional on 1.B3's landing**, decided in §D — this item does not take on standing up a metrics registry as a side effect of its own scope.
- Batch cap number (100) and rate limit number (10/min): **grounded in the edge's verified `LIMIT 100` / 30 s cadence and the verified Force Sync path** (one events-POST per invocation, same query; non-2xx leaves rows untouched and surfaces an operator-visible error), decided in §E — the automatic drain never approaches the limit, and even a force-sync burst at the full ceiling still drains ~60k events/hour.
- Dead-letter retention (90 days) vs the inbox's 365: **decided in §C** — the two tables serve different safety properties (replay-dedup vs. forensic triage) and do not need matching retention windows.
- Testcontainers context strategy: **`@DataJpaTest` slice + `@Import` + `@MockBean`s, not `@SpringBootTest`**, decided in §G — full-context boot fails on `DataInitializer` without super-admin bootstrap credentials and drags in MQTT/WebSocket config the sync path doesn't need.
