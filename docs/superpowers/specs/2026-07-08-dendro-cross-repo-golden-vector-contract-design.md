# Dendro Cross-Repo Golden-Vector Contract (DD5)

**Status:** Draft
**Refactor-program item:** 2.3 (DD5 тАФ "contract, don't deduplicate")
**Focus: both repos** (osi-os owns the fixtures; osi-server mirrors + runs them)
**Depends on:** 2.2 (the `osi-dendro-analytics` module + the first `docs/contracts/dendro/` fixtures it creates). Server-side CI execution additionally depends on 1.B3 (osi-server CI); until 1.B3 lands, the server side runs the fixtures via local `./gradlew test` only, and the osi-os byte-parity gate is the standing cross-repo enforcement.

## Problem

The dendrometer stress/irrigation logic exists in **two** implementations that legitimately differ:
- **Edge** (`osi-os`): the `osi-dendro-analytics` module (extracted in 2.2) тАФ offline JS, computes daily TWD/MDS/stress/irrigation on the Pi from local rows.
- **Cloud** (`osi-server`): `org.osi.server.analytics.DendroDaily` / `DendroAnalyticsService` (Java) тАФ the server-side recompute, including the forecast-driven `TWD_rel` v6 and RDI work.

DD5 rules this a **contract, not a deduplication**: unifying the two couples release cadences and breaks offline-first (the edge must compute without the cloud). The correct control is **shared golden-vector fixtures both repos run in their own frameworks**, so a divergence in the stress/TWD math is **detected** (a failing side names which implementation moved and why), not **prevented** by forcing one codebase. `channels.json` is the single field-name truth both sides consume (DD5).

Today there is no such fixture set and no cross-repo check that the two dendro implementations agree on any input. 2.2 creates the first `docs/contracts/dendro/` fixtures (input rows тЖТ expected edge outputs). This item (2.3) turns those into a **cross-repo contract**: osi-server mirrors the fixtures byte-for-byte and runs them against `DendroDaily`; osi-os CI enforces the byte-mirror; each side asserts its own implementation reproduces the fixture outputs, and a divergence is a documented, attributable failure.

## Verified ground truth

Read directly, 2026-07-08:

1. **osi-server's dendro TWD/envelope compute is `EnvelopeTwd.compute(...)` (a `public static` pure method), orchestrated by `DendroAnalyticsService`; `DendroDaily` is a JPA `@Entity` persistence DTO with NO compute.** (`backend/src/main/java/org/osi/server/analytics/`.) `DendroDaily.java` is where the shared field *names* live as JPA `@Column`s тАФ both camelCase (`twdDayUm`, `mdsUm`, `envelopeRefUm`) and snake_case (`twd_day_um`, `mds_um`, `envelope_ref_um`) тАФ matching the edge node's `dendrometer_daily` columns exactly; the *computation* of those values is `EnvelopeTwd.compute` (envelope/TWD/MDS) called from `DendroAnalyticsService.computeForDevice`. The two sides share the daily-aggregate column vocabulary (via the `dendrometer_daily` schema, not channels.json тАФ ┬з2); the contract pins the *values* the pure compute units produce from the same inputs (┬зC).
2. **`channels.json` covers dendro INPUT (telemetry) fields, NOT the daily-aggregate OUTPUT fields тАФ an important DD5-scoping correction.** `web/react-gui/src/channels/channels.json` (25 channels) contains the per-uplink dendro telemetry fields the daily compute *consumes* тАФ `dendro_position_mm`, `dendro_position_raw_mm`, `dendro_delta_mm`, `dendro_stem_change_um`, `dendro_ratio`, `adc_ch0v`, `adc_ch1v` (all `cardType: dendro`, each with matching `edgeField`/`serverField`). It does **NOT** contain the daily-aggregate outputs `twd_day_um`, `mds_um`, `stress_level`, `envelope_ref_um` (verified absent, 2026-07-08). So DD5's "channels.json as field-name truth" applies to the **module inputs** (the fixture `*.input.json` telemetry rows), while the **module outputs** (`*.expected.json` daily fields) share their vocabulary via the `dendrometer_daily` **table schema**, which both repos already mirror (edge columns == `DendroDaily.java` snake_case per ┬з1). This split is recorded as a DD5 refinement in ┬зOpen decisions тАФ the fixture uses channels.json names for inputs and `dendrometer_daily` column names for outputs, and neither is invented by this item.
3. **osi-server already has substantial dendro JUnit tests, and the shared compute is fixture-drivable AS A PURE UNIT, not via the full service.** Tests: `DendroConfidenceGatingTest` (a plain unit test of pure classification helpers тАФ the precedent that these units test in isolation), `DendroAnalyticsServiceHistoryTest`, `DendroAnalyticsRecomputeRegressionTest`, `DendroCalibrationTest`, plus Agroscope/RDI shadow tests, under `backend/src/test/java/org/osi/server/analytics/`. **The `computeForAllZones()` service entry point is NOT fixture-drivable** тАФ no args, reads `@Autowired` repos, stamps `Instant.now()`, pulls weather from a `WeatherResolver` bean. The fixture-drivable, deterministic, shared-with-edge unit is the `public static EnvelopeTwd.compute(points, method, maxGrowth)` (and the pure helpers `DendroConfidenceGatingTest` already exercises). So the contract runner (┬зC) is a plain JUnit test over those pure units тАФ runnable today via `./gradlew test` **from `backend/`** (the wrapper is `backend/gradlew`; no repo-root wrapper); only CI (the automated runner) is missing.
4. **osi-server has NO CI** (`.github/` holds only `pull_request_template.md` тАФ program ground truth #5, confirmed). So the *automated* server-side fixture run is gated on 1.B3 (which stands up osi-server CI). Until then, the server side is run by a developer via `./gradlew test`, and the **osi-os byte-parity gate is the always-on cross-repo enforcement** (it needs no osi-server CI тАФ it runs in osi-os's existing `migrations.yml` against a read-only osi-server checkout).
5. **The cross-repo checkout + parity precedent is `scripts/verify-sync-op-parity.js`**, wired in `migrations.yml`: osi-os CI checks out `Open-Smart-Irrigation/osi-server` into `osi-server/` via `actions/checkout` with `secrets.OSI_SERVER_RO_TOKEN` (`migrations.yml:18-27`), verifies the checkout is present (fails with a clear `::error::` if the token is unconfigured), and runs a parity script comparing osi-os artifacts against the server source. 2.3's fixture-mirror gate reuses this exact plumbing тАФ same token, same checkout, same "fail if the mirror diverges" shape.
6. **The mirror-into-osi-server convention already exists for a sibling contract:** `docs/contracts/sync-schema/README.md` states "Files here are the source of truth; any mirrored copies in `osi-server` must match bytewise." 2.3 adds `docs/contracts/dendro/` under the same convention and README pattern.
7. **`docs/contracts/dendro/` is created by 2.2**, not this item тАФ 2.2 seeds it with input-rowsтЖТexpected-edge-output fixtures captured from the pre-extraction edge node. 2.3 defines the **cross-repo lifecycle**: the byte-mirror location in osi-server, the osi-os parity gate, the server-side runner, and the divergence-handling rule.

## Design

### A. Fixture format (osi-os is the source of truth)

The contract is scoped to the **envelope-compute layer** both sides expose as a pure function (┬зC): the input is a per-day `DailyPoint` sequence (already-extracted dMax/dMin), the output is the envelope/TWD/MDS values. Each fixture is a JSON pair, shaped so both frameworks parse it identically:

```
docs/contracts/dendro/
  README.md                      # source-of-truth + bytewise-mirror rule + divergence rule (this item adds it)
  cases/
    <case-name>.input.json       # { method: "stepwise"|"linear", maxGrowthUmPerDay: <num>,
                                  #   points: [ { date:"YYYY-MM-DD", d_max_um:<num>, d_min_um:<num>, anchor_eligible:true }, ... ] }
    <case-name>.expected.json    # { results: [ { date:"YYYY-MM-DD", envelope_ref_um:<num>,
                                  #               twd_night_um:<num>, twd_day_um:<num>, mds_um:<num> }, ... ] }
  MANIFEST.json                  # { schemaVersion:<int>, cases:[ "<name>", ... ] }
```

- **Input granularity = `DailyPoint`, not raw telemetry (┬зC boundary).** Each `points[]` entry maps 1:1 to the edge module's `computeEnvelope` sequence element (`{dMax, dMin}` + date) and to the server's `EnvelopeTwd.DailyPoint(date, dMax, dMin, anchorEligible)`. The raw-readingтЖТdMax/dMin extraction is out of *this* contract (the server's `extractExtremes` is service-private); it is covered by the edge's own 2.2 unit tests. This keeps both sides on a pure, deterministic function.
- **Fixture field names (┬зground-truth 2 refined):** input `d_max_um`/`d_min_um`/`date`/`anchor_eligible` and output `envelope_ref_um`/`twd_night_um`/`twd_day_um`/`mds_um` are all `dendrometer_daily` **column names** (the schema both repos mirror; edge columns == `DendroDaily.java`/`EnvelopeTwd.EnvelopeResult` snake_case). At the `DailyPoint`/envelope level the shared vocabulary is the `dendrometer_daily` schema, NOT `channels.json` (which covers raw telemetry channels one layer earlier, and is the field-name truth for the raw-reading layer that this cross-repo contract does not span). Neither name set is invented by this item.
- **No wall-clock, no weather input.** The asserted pure units (`EnvelopeTwd.compute`, edge `computeEnvelope`) take no clock and no weather тАФ determinism is by construction, no stub needed, no `computedAt` field required at this layer. (The `irrDecision`/`computedAt` concern from 2.2 ┬зB belongs to the irrigation-decision layer, which is NOT in this cross-repo contract тАФ only the envelope/TWD/MDS core is.)
- **Numeric tolerance:** expected values are pre-rounded to the shared `round(...)` precision (envelope/TWD/MDS are ┬╡m integers тАФ both sides `round(x, 0)`). The contract is **exact equality on the rounded values**, not float-epsilon тАФ a rounding divergence is a real finding to surface.

### B. osi-os side тАФ run fixtures against `osi-dendro-analytics` (node --test)

A fixture-driven suite, `scripts/test-dendro-contract.js` (`node --test`), loads every `cases/*.input.json`, maps each `points[]` entry to the edge module's `computeEnvelope` sequence element (`{dMax: d_max_um, dMin: d_min_um}` + date), calls `osi-dendro-analytics.computeEnvelope(sequence, method)`, and asserts each result element's `{envelopeRef, twdNight, twdDay, mds}` equals the case's `expected.results[i]` `{envelope_ref_um, twd_night_um, twd_day_um, mds_um}` (exact equality on the rounded ┬╡m integers). This is the edge side of the contract: **the edge `computeEnvelope` must reproduce its committed fixtures**. Wired into `migrations.yml` (Node 22).

The fixtures are captured from the edge `computeEnvelope` at the `DailyPoint` level (2.2's harness snapshots the module's envelope output over a multi-day sequence тАФ see ┬зA's multi-day-anchor note below), so this suite is green at introduction; its ongoing value is a regression tripwire: any future edit to `computeEnvelope` that moves an envelope/TWD/MDS value fails here with the exact case + field.

**Fixtures must span a multi-day anchor sequence to be non-vacuous.** A single-`DailyPoint` case degenerates (`envelopeRef == dMax`, `twdNight == 0`) тАФ the envelope's whole point is the running-max anchor reset across days. Each case's `points[]` therefore carries several days (e.g. a growth day that resets the anchor upward, then deficit days where `envelopeRef` holds and `twdNight`/`twdDay` grow), so the contract exercises the anchor logic both implementations share.

### C. osi-server side тАФ mirror + run fixtures against the server's PURE compute units

**The runner targets the server's pure static compute units, NOT the repository-driven service** тАФ the load-bearing correction from verification (see ┬зground-truth 3). `DendroAnalyticsService.computeForAllZones()` is **not** a fixture-drivable entry point: it takes no arguments, reads all inputs from `@Autowired` JPA repositories, stamps `Instant.now()` internally, and pulls weather from an injected `WeatherResolver` bean тАФ driving it from a fixture would require a full `@SpringBootTest` seeded into Postgres, and it would still be non-deterministic (`Instant.now()`), defeating ┬зA's exact-equality contract. `DendroDaily` is a JPA `@Entity` (persistence DTO) with **no compute at all**.

The genuinely shared, fixture-drivable compute is the **pure static envelope unit** the service delegates to: `EnvelopeTwd.compute(List<DailyPoint>, method, maxGrowthUmPerDay)` (a `public static` method returning `List<EnvelopeResult>`, no Spring context, no clock, no weather), the exact server-side counterpart of the edge module's `computeEnvelope`. **The contract's input level is per-day `DailyPoint`s (`date, dMax, dMin, anchorEligible`), NOT raw telemetry readings** тАФ this is load-bearing (see below): `EnvelopeTwd.compute` takes already-extracted per-day dMax/dMin, and the raw-readingтЖТdMax/dMin extraction (`extractExtremes`) is a **`private` method of `DendroAnalyticsService`** on the server, not a pure public unit. So the cross-repo contract is scoped to the layer both sides expose purely тАФ the envelope/TWD/MDS compute over `DailyPoint`s тАФ and both sides are pure and deterministic by construction. The forecast-driven full-pipeline path (`computeForAllZones` тЖТ `weatherResolver` тЖТ `TWD_rel` v6/RDI) is explicitly **out of the shared contract**.

**Why the input is `DailyPoint`s, not raw readings тАФ the extraction-layer boundary.** On the edge, `extractExtremes` (raw readings тЖТ per-day dMax/dMin) IS in the extracted pure module (2.2 ┬зA) and IS unit-tested there (2.2 ┬зD-1). On the server, the equivalent `extractExtremes` is `private` to `DendroAnalyticsService` (verified) тАФ driving it from a fixture would require the same infeasible full-service/Spring path ┬зC rejects. Making the cross-repo contract start from `DailyPoint`s draws the boundary exactly at the layer both sides expose as a pure function, so the contract stays honest (no server refactor demanded) and still pins the physiologically-meaningful envelope/TWD/MDS math тАФ the part most worth a cross-repo tripwire. The edge's rawтЖТdMax/dMin extraction is covered by the edge's own 2.2 unit tests; it is not part of *this cross-repo* contract because the server does not expose it purely. (If a future item makes the server's extraction a pure unit, a raw-reading-level contract layer can be added then.)

**Mirror location:** `osi-server/backend/src/test/resources/contracts/dendro/` (test resources, read off the classpath), byte-identical to `docs/contracts/dendro/cases/` + `MANIFEST.json`. The README's rule (┬зground-truth 6) applies: osi-os is source of truth; the server copy must match bytewise.

**Runner:** a new plain JUnit test in `backend/src/test/java/org/osi/server/analytics/` (e.g. `DendroContractFixtureTest`) тАФ **no Spring context** тАФ that, for each case in `MANIFEST.json`, maps each `points[]` entry to `new EnvelopeTwd.DailyPoint(LocalDate.parse(date), d_max_um, d_min_um, anchor_eligible)` (the record is `DailyPoint(LocalDate date, double dMax, double dMin, boolean anchorEligible)` тАФ note `anchorEligible`, the slot the service fills with `!lowConfidence`), calls `EnvelopeTwd.compute(points, method, maxGrowthUmPerDay)`, and asserts each `EnvelopeResult`'s `envelopeRef()`/`twdNight()`/`twdDay()`/`mds()` equals the case's `expected.results[i]` `envelope_ref_um`/`twd_night_um`/`twd_day_um`/`mds_um`. (`EnvelopeResult` is a 10-field record; only these four accessors are asserted.) Runs via `./gradlew test` **from `backend/`** (the Gradle wrapper is `backend/gradlew`, no repo-root wrapper); runs in CI once 1.B3 lands. Because both `computeEnvelope` (edge) and `EnvelopeTwd.compute` (server) are pure and take pre-extracted `DailyPoint`s, the runner needs no DB, no weather bean, and no clock тАФ matching ┬зA's determinism requirement without a stub, and starting both sides at the identical input so the readingsтЖТextremes divergence (which the server does not expose purely) never enters the contract.

**The asserted shared field set (the intersection тАФ divergence DETECTED not prevented, and attributed).** The contract asserts only fields both sides compute purely from the same historical rows: the envelope/TWD/MDS core тАФ `envelope_ref_um`, `twd_day_um`, `twd_night_um`, `mds_um` (all present in both `EnvelopeTwd.EnvelopeResult`/`dendrometer_daily` and the edge `computeEnvelope` output). Fields the server derives through the forecast/`TWD_rel`-v6/RDI path (`twd_rel`, RDI outputs, `sd_vpd_r2`) are server-only extensions, **excluded** as legitimate DD5 divergence. **`stress_level` is EXCLUDED from the asserted intersection** in this item: the server's stress classification is `TWD_rel`-v6-influenced (`tree_state_v5` vs `stress_level` divergence in `DendroDaily`), so asserting it would produce legitimate-but-noisy reds that are DD5-expected divergence, not regressions тАФ the shared core is the raw TWD/MDS/envelope math, and stress classification is where the two implementations *intentionally* differ. (A later item may add a separately-scoped stress-classification sub-contract if the two ever converge; not this item.) This keeps the contract non-vacuous (four real shared numeric fields) without manufacturing false divergence. The result:
- **Green** when both implementations agree on the shared envelope/TWD/MDS core.
- **Red on exactly one side** when that side's shared-core math changes тАФ the failing side's diff (case + field + edge-vs-server value) documents which implementation moved. Remedy is a human decision (fix the regression, or update the fixture on the side that legitimately moved), never automated reconciliation.

### D. osi-os cross-repo byte-parity gate (the always-on enforcement)

A new osi-os CI verifier, `scripts/verify-dendro-contract-mirror.js`, wired into `migrations.yml` alongside `verify-sync-op-parity.js`, reusing the same osi-server read-only checkout (┬зground-truth 4):
- For every file under `docs/contracts/dendro/cases/` + `MANIFEST.json`, assert a byte-identical copy exists at the mirror path in the checked-out `osi-server/backend/src/test/resources/contracts/dendro/`.
- Fail with a clear message naming any missing or byte-diverging file (mirroring `verify-sync-op-parity.js`'s failure style, and the checkout-missing `::error::` guard).
- This gate needs **no** osi-server CI тАФ it runs in osi-os's existing workflow against the read-only checkout, so the byte-mirror is enforced from day one; the server-side *value* check (┬зC) joins automated enforcement when 1.B3 lands. Until then, a PR that changes a fixture in osi-os without updating the osi-server mirror fails this osi-os gate immediately, keeping the two copies from drifting even before osi-server has CI.

**Delivery direction:** osi-os тЖТ osi-server (osi-os is source of truth). Because osi-os CI cannot *write* to osi-server, updating the mirror is a coordinated two-PR change (osi-os fixture change + osi-server mirror update); the byte-parity gate turns a forgotten mirror update into a red osi-os build. This is the same manual-mirror discipline `sync-schema` already uses (┬зground-truth 5) тАФ 2.3 does not build automated cross-repo copy (that is out of scope and YAGNI at two repos).

### E. Field-name truth: `channels.json` (inputs) + `dendrometer_daily` schema (outputs) тАФ DD5, cite item 2.5

DD5 names `channels.json` the single field-name truth consumed by both builds; program item **2.5** exports it into the osi-server build. 2.3 does **not** re-derive or duplicate that. The correction from ┬зground-truth 2 is load-bearing here: `channels.json` is the truth for the **module inputs** тАФ the fixture `*.input.json` telemetry rows use its dendro channel names (`dendro_position_mm`, `dendro_ratio`, `adc_ch0v`, тАж). The **module outputs** (`*.expected.json` daily-aggregate values: `twd_day_um`, `mds_um`, `stress_level`, тАж) are NOT in `channels.json` today; their shared vocabulary is the `dendrometer_daily` table schema, which the edge (`dendrometer_daily` columns) and server (`DendroDaily.java` snake_case, ┬зground-truth 1) already mirror. So 2.3 uses two existing truth sources by direction, invents neither. If 2.5 lands first and *extends* `channels.json` to cover daily-aggregate fields, the ┬зA parity check can validate both directions against it; if not, the daily-output names remain schema-anchored. **2.3 is not gated on 2.5**, and does not require `channels.json` to grow тАФ it consumes what each side already agrees on.

## Non-goals

- **Unifying the two dendro implementations** тАФ DD5 forbids it; the contract detects divergence, it does not eliminate the second implementation.
- **Asserting server-only fields** (`TWD_rel` v6, RDI, `sd_vpd_r2`) or **`stress_level`** against the edge тАФ those are legitimate server extensions / intentional-divergence points outside the shared contract; the asserted set is the pure envelope/TWD/MDS core only (`envelope_ref_um`, `twd_day_um`, `twd_night_um`, `mds_um`) (┬зC).
- **Driving the server's full `computeForAllZones` pipeline from a fixture** тАФ infeasible (no-arg, repo-backed, clock- and weather-driven); the contract targets the pure static units (`EnvelopeTwd.compute` + pure helpers), not the service or the JPA entity (┬зC).
- **Building automated cross-repo fixture copy** тАФ the mirror is manual + byte-parity-gated, matching `sync-schema`; automated copy is YAGNI at two repos.
- **Standing up osi-server CI** тАФ that is 1.B3; 2.3's server-side automated run depends on it, and until then runs via local `./gradlew test` (from `backend/`) with the osi-os byte-parity gate as the standing enforcement.
- **Creating the fixtures** тАФ 2.2 creates `docs/contracts/dendro/` and the first cases; 2.3 defines the format contract, the mirror, the gates, and the runners.
- **Changing `channels.json`** тАФ 2.3 consumes it; 2.5 exports it.
- **Any live gateway / production host / SSH.**

## Open decisions

None outstanding.

- **Fixture field names split by direction: inputs = `channels.json` dendro channels; outputs = `dendrometer_daily` columns** тАФ ┬зA/┬зE, ┬зground-truth 2; channels.json covers telemetry inputs (not the daily-aggregate outputs), so the two directions draw their shared vocabulary from different existing sources, neither invented here.
- **Server-side runner targets the pure static units (`EnvelopeTwd.compute` + pure helpers), not `computeForAllZones` or the `DendroDaily` entity** тАФ ┬зC, ┬зground-truth 1/3; the service entry point is repo/clock/weather-driven and not fixture-drivable, while the pure units are deterministic by construction.
- **Contract asserts the envelope/TWD/MDS intersection only; `stress_level` + server-only forecast/RDI fields excluded** тАФ ┬зC; `stress_level` classification is `TWD_rel`-v6-influenced on the server (intentional divergence), so asserting it would manufacture false reds; the four raw numeric fields keep the contract non-vacuous.
- **Exact equality on rounded values, not float tolerance** тАФ ┬зA; both sides share rounding rules, so a rounding divergence is a real finding to surface.
- **Mirror is manual + byte-parity-gated in osi-os CI (no automated cross-repo copy)** тАФ ┬зD; matches `sync-schema`, YAGNI at two repos, and works before osi-server has CI.
- **Server-side automated run gated on 1.B3; osi-os byte-parity gate is the standing enforcement meanwhile** тАФ ┬зC/┬зD, ┬зground-truth 3.
- **No weather stub needed; the forecast path is excluded, not stubbed** тАФ ┬зA/┬зC; the server's weather comes from an injected `WeatherResolver` bean (not an input row), and the asserted pure envelope/TWD/MDS units never touch weather, so both sides are deterministic without any stub. (Corrects an earlier framing that weather would be an explicit fixture field.)
