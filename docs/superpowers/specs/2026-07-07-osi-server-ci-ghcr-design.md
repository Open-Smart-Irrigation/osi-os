# osi-server CI + ArchUnit Boundary Test + Micrometer Metrics + GHCR Publish + Pull-Only VPS Deploys

**Status:** Draft
**Refactor-program item:** 1.B3 (DD11 + DD16)
**Focus: osi-server**
**Runway:** this item is the runway for 1.B4's Testcontainers tests (already specced: they need Docker-capable CI runners); removes the on-host-build production hazard.

## Problem

`osi-server` has zero CI. `.github/` holds only a PR template. 159 test files exist (Mockito-repository style), but nothing runs them on push or PR — a broken build can merge silently. Worse, the production VPS builds its own Docker images: `docker/docker-compose.yml`'s `backend` service runs `docker/backend/Dockerfile`, a multi-stage build compiling two Vite frontends (Node 20) and a Gradle Spring Boot jar, **on the 4 CPU / 4 GB host that also runs Postgres, Mosquitto, Mongo, and the prediction services next to irreplaceable farm-mirror data.** This is the documented cause of production unresponsiveness (`AGENTS.md`: "full on-host rebuilds have made it unresponsive"). There is also no server-side observability beyond a bare `/actuator/health`, and no boundary enforcement on a 20-package monolith that contains a 93 KB god-file (`EdgeSyncService.java`, verified 1817 lines) already flagged for a future split (DD12, item 3.4).

### Contradiction requiring a design correction: DD11's example rule is false today

DD11's charter line and the expert's D1 both propose, as an illustrative rule, "`sync` must not depend on `analytics`." **This is false today**, verified directly: `EdgeSyncService.java` imports 13 classes from `org.osi.server.analytics` (`DendroDaily`, `DendroDailyRepository`, `ChameleonReading`, `ChameleonReadingRepository`, `DendroReading`, `DendroReadingRepository`, `IrrigationEvent`, `IrrigationEventRepository`, `WeatherSource`, `ZoneEnvironmentService`, `ZoneEnvironmentSummary`, `ZoneDailyEnvironment`, `ZoneDailyEnvironmentRepository`, `ZoneDailyRecommendation`, `ZoneDailyRecommendationRepository`) — sync writes analytics-owned domain rows directly, which is architecturally exactly what a narrow-waist future split (DD12) would change, but is today's reality.

**Worse: the codebase already has real package cycles — far more than first sampled.** An initial DFS pass over the import graph surfaced six cyclic pairs; **exhaustive verification (all mutual-import pairs enumerated + Tarjan SCC over the full `import org.osi.server.<pkg>` graph, spot-confirmed by reading actual import lines — e.g. `command.CommandService` → `mqtt.MqttPublisherService` / `mqtt.MqttMessageRouter` → `command.CommandService`; `retention.SyncInboxRetentionJob` → `sync.SyncInboxRepository` / `sync.SyncHealthService` → `retention.DbHealthCounters`) corrects that undercount: there are 15 mutually-importing package pairs** (`analysis↔history`, `analytics↔device`, `analytics↔zone`, `command↔device`, `command↔mqtt`, `command↔user`, `device↔telemetry`, `device↔user`, `device↔zone`, `mqtt↔user`, `prediction↔soil`, `prediction↔zone`, `retention↔sync`, `security↔user`, `soil↔zone`) **and the cyclic core is one 12-package strongly-connected component** (`analytics, command, device, gateway, mqtt, prediction, security, soil, telemetry, user, websocket, zone`) plus the `analysis↔history` and `retention↔sync` two-node SCCs. Only `chameleon`, `channels`, and `config` are cycle-free. This kills any hand-maintained `ignoreDependency(...)` exception list: enumerating a correct feedback-edge set over a 12-package SCC by hand is neither verifiable nor maintainable.

**Decision: the ArchUnit ruleset in this item cannot include a blanket "no cycles among top-level packages" rule or a literal "`sync` must not depend on `analytics`" rule — both fail on today's `main`, and DD11's own charter says these tests must lock what's true, not what's aspirational.** See §B for the ruleset that actually passes today (the cycle rule ships as a `FreezingArchRule` with a committed baseline store — ArchUnit's purpose-built mechanism for exactly this legacy-adoption case). This is a finding for the program map: DD11's illustrative rule text needs updating in `docs/architecture/refactor-program-2026.md` once this spec is reviewed, and the real cycle-breaking work (untangling the 12-package core) is a separate, larger future item — almost certainly the DD12 `EdgeSyncService` split plus sibling work, not something this CI item does. Flagged for the reviewer, not silently downgraded.

## Design

### A. CI workflow shape

Two new GitHub Actions workflows in `osi-server/.github/workflows/`, mirroring osi-os's existing style (`runs-on: ubuntu-latest`, `actions/checkout@v4` with `persist-credentials: false`, pinned tool-setup actions, one `run:` step per check) rather than inventing a new house style:

**`backend-ci.yml`** — triggers `push: [main]` + `pull_request: [main]` (matching `migrations.yml`'s trigger shape):
- `actions/setup-java@v4` with `distribution: temurin`, `java-version: '21'` (the JDK the Dockerfile's `backend-builder` stage uses; Gradle's toolchain resolution will still compile at source level 17), `cache: gradle`.
- `./gradlew test --no-daemon` — per verification 5, this runs unit + the future Testcontainers tests **without** touching the frontend build or needing a Node toolchain in this job. GitHub-hosted `ubuntu-latest` runners have Docker pre-installed and running, which is exactly what 1.B4's `@Testcontainers` Postgres tests need — **this is the concrete "Docker available for future Testcontainers" requirement**; no extra service-container YAML is needed since Testcontainers manages its own container lifecycle against the runner's Docker daemon.
- No `-x` flags: verification 5 established `test` doesn't pull in the frontend tasks, so there is nothing to exclude.

**`prediction-ci.yml`** — same triggers, separate job (independent failure domain, different toolchain):
- `actions/setup-python@v5`, `python-version: '3.11'` (match whatever `prediction-service`'s Dockerfile base uses — confirm exact pin at implementation time by reading `docker/prediction-service/Dockerfile`, not guessed here since it wasn't in this pass's required reading).
- `pip install -r requirements.txt` then `pytest` from `prediction-service/`. Cheap (verification 9): no Docker, no DB, pure unit-level.

**Frontend build is deliberately NOT a separate CI job in this item.** The two Vite builds are exercised for real inside the GHCR image-build workflow (§4) via the same Dockerfile stages that already build them today — building them twice (once bare in CI, once again inside the image build) would double the cost for no new coverage, since there is no frontend test suite invoked here (`AGENTS.md` lists `npm run dev`/`npm run build` for `frontend/`, and separately `terra-intelligence && npm test && npm run build` — **note**: Terra has an `npm test` step AGENTS.md documents but this item does not add a CI job for it, because it's frontend-owned and out of this item's charter line, which names backend CI + ArchUnit + Micrometer + GHCR + deploy, not frontend test wiring. Flag as a candidate follow-up, not built here.)

**Op-parity relationship: left one-sided, by design, not by oversight.** The cross-repo gate today lives only in osi-os's `migrations.yml`, which checks out `osi-server` read-only (via `OSI_SERVER_RO_TOKEN`) and runs `verify-sync-op-parity.js` against `EdgeSyncService.java`. Mirroring this server-side (osi-server checking out osi-os to re-run the same parity check) would require **a second secret** (an osi-os read token, though osi-os is public so this is just `actions/checkout` with a public repo — actually simpler than the reverse direction) but would duplicate an already-working gate for zero new coverage: the parity check is symmetric in what it verifies (do both sides agree on the op set), so running it from either side catches the same drift. **Decision: leave it one-sided.** The existing osi-os-side gate already fails the *osi-os* PR when a sync op is added edge-side without a matching server handler; the gap it does NOT cover is a osi-server-only PR that changes `EdgeSyncService`'s op switch without any osi-os PR — that PR would merge on osi-server with no parity check at all. This is a real gap, but closing it duplicates infrastructure for a change pattern the program's own phasing predicts as rare (op additions are edge-driven today; MClimate in Phase 3 will be the next one, and it touches both repos in the same work item per the existing pattern). Revisit if osi-server-only op changes become common; not worth a second checkout+token pair today.

### B. ArchUnit boundary test (DD11) — scoped to what's true today

One plain JUnit 5 test class, `ArchitectureTest`, in the normal `backend/src/test/java/org/osi/server/` test source set (no new source set, no new Gradle plugin beyond the `archunit-junit5` test dependency). Rules, each verified against §"Contradiction" above to actually pass on current `main`:

1. **No cycles rule is NOT a blanket top-level-package rule** (the cyclic core is a 12-package SCC + two 2-node SCCs, per verification above — a bare rule fails immediately, and a hand-maintained `ignoreDependency(...)` list over that core is untenable). Instead: **`slices().matching("org.osi.server.(*)..").should().beFreeOfCycles()` wrapped in `FreezingArchRule.freeze(...)`** with the violation store (a plain text baseline file) committed under `backend/src/test/resources/archunit-store/`. This is ArchUnit's purpose-built legacy-adoption mechanism and is the exact DD3 baseline-file ratchet pattern from the edge program: existing violations are frozen (grandfathered), any **new** cycle-participating dependency fails the build, and violations that get fixed are automatically removed from the store (the ratchet only tightens). Note the honest strictness consequence: freeze is stricter than "no new cycles" — a new dependency that *grows an existing* frozen cycle also fails, which matches DD3's "may only decrease" spirit; the escape hatch for an intentionally accepted new violation is a reviewed, deliberate edit of the committed store file in the same PR, visible in diff. `freeze.store.default.allowStoreCreation` is set `true` only for the initial store creation, then flipped to `false` so a deleted store fails loudly instead of silently re-freezing. A comment in the test points at this spec section as the record of why the baseline exists.
2. **`sync` must not depend on `analytics` is dropped as written** (false today, per verification) and replaced with the rule that actually reflects the intended boundary direction: **`noClasses().that().resideInAPackage("..analytics..").should().dependOnClassesThat().resideInAPackage("..sync..")`** — this direction passes today (verified: `analytics` has zero imports from `sync`) and is the one that matters for the DD12 split anyway (analytics must not reach back into sync; sync reaching into analytics is the acknowledged, ticketed direction of today's coupling that DD12 will address by extraction, not by this test).
3. **Controllers only in `*Controller` classes**: `classes().that().areAnnotatedWith(RestController.class).or().areAnnotatedWith(Controller.class).should().haveSimpleNameEndingWith("Controller")`. Verified passing: the one non-`*Controller` class matching a controller-shaped annotation is `GlobalExceptionHandler`, which is `@RestControllerAdvice` — a different annotation, correctly excluded by matching only `@RestController`/`@Controller` and not `@RestControllerAdvice`/`@ControllerAdvice`.
4. **No field/constructor injection ban, no layering rule beyond the above** — deliberately minimal. DD11's own framing is "lock boundaries you already have before you refactor," not "invent boundaries you wish existed." A fourth rule (e.g., "repositories only accessed from same-package services") was considered and rejected: 20 packages already cross-call repositories directly in several places (`EdgeSyncService` alone touches ~10 repositories across packages) and asserting otherwise would immediately fail, requiring exactly the kind of pre-emptive refactor DD11 says not to do yet.

Runs as part of `./gradlew test` (no separate Gradle task, no separate CI step) — it is a JUnit test like any other, picked up automatically, and gets Docker-capable-runner CI for free from §A.

### C. Micrometer / actuator metrics

- Add `implementation("io.micrometer:micrometer-registry-prometheus")` to `build.gradle.kts`.
- `application.yml`: extend `management.endpoints.web.exposure.include` from `health,info` to `health,info,prometheus`.
- **Protection decision: `/actuator/prometheus` is NOT publicly routed through Caddy.** Grounded in verification 8 (today's `SecurityConfig` already treats every actuator path except `/actuator/health` as behind the normal auth chain — this item **preserves that existing posture** rather than inventing a new one) and verification 3 (`caddy-net` is the only route in from outside the compose network — the `osi-internal` network is not exposed to Caddy or the host at all today, meaning any container on `osi-internal` alone, without `caddy-net`, is already unreachable from outside the Docker host). Because `backend` is on **both** networks (it needs `caddy-net` for its own reverse-proxied traffic), the endpoint isn't network-isolated for free the way a second internal-only service would be — so this item adds an explicit `SecurityConfig` rule: `/actuator/prometheus` requires `hasRole("ADMIN")` (or `SUPER_ADMIN`), same pattern as the existing `/api/v1/admin/**` surface, rather than `permitAll` like `/actuator/health`. No Caddy config change is proposed (Caddy is not owned by this repo and touching it is out of scope) — the admin-role gate is the actual protection, and it composes safely regardless of whether an operator later chooses to also block the path at the Caddy layer.
- **JVM heap cap:** add `JAVA_TOOL_OPTIONS: "-Xmx768m"` to the `backend` service's `environment:` block in `docker-compose.yml`, alongside the existing `JAVA_TOOL_OPTIONS` entry (today it carries only `-Dosi.rate-limit.trusted-proxies=...` — this item appends `-Xmx768m` to the same string, doesn't add a second env var). **768 MB, not the expert's upper bound of ~1 GB**, because the shared 4 GB host (per `AGENTS.md`, confirmed) runs Postgres 16 + Mosquitto + Mongo 7 (OpenAgri) + `openagri-weather` + `prediction-service` + `fao-reference-service` alongside the backend — six other long-running containers on the same box, several of which (Postgres, Mongo) want their own meaningful working-set memory. Choosing the low end of the expert's ~768 MB–1 GB range leaves more headroom for the DB layer the sync/ingest path depends on, at the cost of the JVM GCing sooner under load; this is exactly the kind of number that should be watched via the new Prometheus JVM metrics (`jvm_memory_used_bytes`) and revised with evidence rather than guessed twice. Scrape/alerting setup itself (Prometheus server, Grafana) is out of scope — this item only makes the numbers *available*.

### D. GHCR publish

- **Image name:** `ghcr.io/open-smart-irrigation/osi-server-backend` (org-scoped, matches the private-repo owner exactly — GHCR ties default visibility to the pushing org/repo, per verification 7).
- **Visibility: private**, matching the source repo (verification 7). No justification exists for a public image of a private codebase's build artifact.
- **Tagging: immutable `sha-<short>` (7-char) on every build + a floating `latest` on `main` builds only.** `sha-<short>` gives the rollback runbook (§F) an unambiguous, never-reused reference — the exact image that was built from a specific commit, needed for "redeploy the previous tag" to mean something concrete. `latest` is kept purely as a convenience default for a manual first pull / local testing, never used as the deploy-time tag in the runbook (the runbook always names a `sha-<short>` explicitly) — this avoids the classic `latest`-drift failure mode (two different hosts silently running different code under the same tag).
- **Build workflow (`ghcr-publish.yml`):** triggers on push to `main` only (not PRs — an image build on every PR would burn CI minutes and GHCR storage for code that hasn't merged; the existing `backend-ci.yml` already gates PRs on tests). The workflow declares an explicit `permissions: { contents: read, packages: write }` block — `GITHUB_TOKEN`'s default permissions can be read-only depending on org settings, and the first push also creates the package namespace. Job: `docker/setup-buildx-action@v3`, `docker/login-action@v3` against `ghcr.io` using the built-in `${{ secrets.GITHUB_TOKEN }}` (GHCR accepts the repo's own Actions token for pushing to that repo's package namespace — no new secret needed for the *publish* side), `docker/build-push-action@v5` with `context: .`, `file: docker/backend/Dockerfile`, `platforms: linux/amd64` (single-arch, per verification 10), `tags: ghcr.io/open-smart-irrigation/osi-server-backend:sha-${{ steps.sha.outputs.short }},ghcr.io/open-smart-irrigation/osi-server-backend:latest`, and the existing `VITE_MAPBOX_TOKEN` build arg sourced from a repo secret (it's already a build arg today, per the Dockerfile — this item does not change how it's supplied, only where the build runs).
- **Layered bootJar:** confirm at implementation time that the produced jar has a `BOOT-INF/layers.idx` (Spring Boot 3.4.3 default). If confirmed present, no `build.gradle.kts` change is needed — layering already happens; if absent (e.g., disabled somewhere not found in this pass), add the minimal `tasks.named<org.springframework.boot.gradle.tasks.bundling.BootJar>("bootJar") { layered { enabled = true } }` block. Either way, the Dockerfile's runtime stage should extract layers (`java -Djarmode=layertools -jar app.jar extract` in an intermediate step, or keep the current single-`COPY … app.jar` shape if the marginal pull savings don't justify a Dockerfile rewrite this item — **decision: keep the current single-jar `COPY` for this item's first cut**, since the VPS's actual bottleneck today is CPU/RAM during *build*, not the pull's byte count over its network link; layered extraction is a cheap follow-up once GHCR pulls are the steady-state and the marginal win becomes worth measuring, not a blocker for shipping the bigger safety change now).
- **VPS pull credential:** a **read-only PAT (classic, `read:packages` scope only, or a fine-grained token scoped to `Open-Smart-Irrigation/osi-server` package read)**, generated by the operator (Phil) outside any repo, stored **only** in the VPS's `.env` (new var `GHCR_PULL_TOKEN`, documented in `.env.example` as a placeholder, never a real value) and used for a one-time `docker login ghcr.io -u <github-username> -p "$GHCR_PULL_TOKEN"` on that host. **Never committed, never logged, never in a GitHub Actions secret consumed by any workflow this item adds** (the publish side uses `GITHUB_TOKEN`, which is unrelated and scoped only to that job's run). This is the same "ephemeral/scoped credential, never in repo docs or SSH config" policy already governing production access per project memory — applied here to a new credential class the same way.

### E. Compose cutover

- **`backend` service switches `build:` → `image: ghcr.io/open-smart-irrigation/osi-server-backend:${BACKEND_IMAGE_TAG:-latest}`** (new `.env` var, defaulting to `latest` for a fresh non-production clone that hasn't set up GHCR auth yet — local dev users who just want `docker compose up` to work with the previous on-host `build:` behavior lose nothing if they instead run a documented `docker compose -f docker-compose.yml -f docker-compose.dev-build.yml` override that restores `build:` for local iteration; that dev override file is a one-line addition, not built in this pass since it's not blocking the safety fix).
- **`prediction-service` and `fao-reference-service` switch too, in this same item** — not deferred. Justification from verification 1: both already build cleanly from `docker/prediction-service/Dockerfile` with named targets (`runtime`, `fao-runtime`) that don't depend on anything VPS-specific (no GPU, no host-mounted model cache in the *default* compose — that's only the separate `docker-compose.vps.yml` override for the Chronos-2 GPU variant, out of scope, see below), and a `prediction-ci.yml` job (§A) already proves the Python code imports/runs; adding a second `docker/build-push-action@v5` step for each target in the same `ghcr-publish.yml` workflow is marginal cost once the workflow and GHCR auth exist. Leaving them on `build:` while fixing only `backend` would leave the VPS still compiling two non-trivial Docker images (a Python FastAPI service with ML dependencies is not free to build either) next to the same farm-mirror data — defeating half the point of this item. **`prediction-validation` stays on `build:`** — it's `profiles: ["validation"]`-gated, never runs in a plain `docker compose up -d`, and is explicitly a one-shot validation harness, not steady-state production capacity; publishing it to GHCR would be dead infrastructure for a path that's invoked manually and rarely.
- **`mosquitto` stays on `build:`** — its Dockerfile is a ~10-second Alpine layer (verification 1); publishing it to GHCR adds a fourth image/tag pair to track for a build that isn't the hazard this item exists to close. Explicitly deferred, not silently left as an oversight: if `mosquitto`'s build ever grows (it hasn't, and there's no reason it would), revisit.
- **`docker-compose.vps.yml` (the Chronos-2/GPU prediction-service override, verified to exist under `docker/prediction-service/`) is untouched by this item.** It's a separate, `-f`-selected override file with its own `build:` block and a HuggingFace model cache volume — a materially different (and heavier, ML-model-download-dependent) build than the default `prediction-service` target this item publishes. Converting it is a larger, separate follow-up (its build likely can't run on a plain GitHub-hosted runner without a much longer job and larger image), noted here as an explicit deferral, not a silent gap: **the default `docker-compose.yml` path (what `AGENTS.md`'s documented `docker compose up -d` uses) is fully covered by this item; the GPU override is not.**
